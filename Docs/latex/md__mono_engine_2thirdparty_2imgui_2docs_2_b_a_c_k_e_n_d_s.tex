💡 The {\bfseries{\href{https://github.com/ocornut/imgui/wiki/Getting-Started}{\texttt{Getting Started}} wiki guide}} has examples of how to integrate Dear Im\+Gui into an existing application. ~\newline
 The \href{https://github.com/ocornut/imgui/blob/master/docs/EXAMPLES.md}{\texttt{EXAMPLES.\+MD}} documentation may also be worth a read.\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md516}{}\doxysubsubsection{\texorpdfstring{What are Backends?}{What are Backends?}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md516}
Dear Im\+Gui is highly portable and only requires a few things to run and render, typically:\+


\begin{DoxyItemize}
\item Required:\+ providing mouse/\+keyboard inputs (fed into the {\ttfamily \doxylink{struct_im_gui_i_o}{Im\+Gui\+IO}} structure).
\item Required:\+ creating, updating and destroying textures.
\item Required:\+ rendering indexed textured triangles with a clipping rectangle.
\end{DoxyItemize}

Extra features are opt-\/in, our backends try to support as many as possible:\+


\begin{DoxyItemize}
\item Optional:\+ custom texture binding support.
\item Optional:\+ clipboard support.
\item Optional:\+ gamepad support.
\item Optional:\+ mouse cursor shape support.
\item Optional:\+ IME support.
\item Optional:\+ multi-\/viewports support. etc.
\end{DoxyItemize}

This is essentially what each backend is doing + obligatory portability cruft. Using standard backends ensure you can get all those features including the ones that would be harder to implement on your side (e.\+g. multi-\/viewports support).

It is important to understand the difference between the core Dear Im\+Gui library (files in the root folder) and the backends which we are describing here (backends/\+ folder).


\begin{DoxyItemize}
\item Some issues may only be backend or platform specific.
\item You should be able to write backends for pretty much any platform and any 3D graphics API. e.\+g. you can get creative and use software rendering or render remotely on a different machine.
\end{DoxyItemize}\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md517}{}\doxysubsection{\texorpdfstring{Using standard Backends}{Using standard Backends}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md517}
{\bfseries{The \href{https://github.com/ocornut/imgui/blob/master/backends}{\texttt{backends/\+}} folder contains backends for popular platforms/\+graphics API, which you can use in your application or engine to easily integrate Dear Im\+Gui.}} Each backend is typically self-\/contained in a pair of files:\+ imgui\+\_\+impl\+\_\+\+XXXX.\+cpp + imgui\+\_\+impl\+\_\+\+XXXX.\+h.


\begin{DoxyItemize}
\item The \textquotesingle{}Platform\textquotesingle{} backends are in charge of:\+ mouse/\+keyboard/\+gamepad inputs, cursor shape, timing, and windowing.~\newline
 e.\+g. Windows (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_win32.cpp}{\texttt{imgui\+\_\+impl\+\_\+win32.\+cpp}}), SDL3 (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_sdl3.cpp}{\texttt{imgui\+\_\+impl\+\_\+sdl3.\+cpp}}), GLFW (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_glfw.cpp}{\texttt{imgui\+\_\+impl\+\_\+glfw.\+cpp}}), etc.
\item The \textquotesingle{}Renderer\textquotesingle{} backends are in charge of:\+ creating atlas texture, and rendering imgui draw data.~\newline
 e.\+g. Direct\+X11 (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_dx11.cpp}{\texttt{imgui\+\_\+impl\+\_\+dx11.\+cpp}}), Open\+GL/\+\+Web\+GL (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_opengl3.cpp}{\texttt{imgui\+\_\+impl\+\_\+opengl3.\+cpp}}), Vulkan (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_vulkan.cpp}{\texttt{imgui\+\_\+impl\+\_\+vulkan.\+cpp}}), etc.
\item For some high-\/level frameworks, a single backend usually handles both \textquotesingle{}Platform\textquotesingle{} and \textquotesingle{}Renderer\textquotesingle{} parts.~\newline
 e.\+g. Allegro 5 (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_allegro5.cpp}{\texttt{imgui\+\_\+impl\+\_\+allegro5.\+cpp}}). If you end up creating a custom backend for your engine, you may want to do the same.
\end{DoxyItemize}

An application usually combines one Platform backend + one Renderer backend + main Dear Im\+Gui sources. For example, the \href{https://github.com/ocornut/imgui/tree/master/examples/example_win32_directx11}{\texttt{example\+\_\+win32\+\_\+directx11}} application combines imgui\+\_\+impl\+\_\+win32.\+cpp + imgui\+\_\+impl\+\_\+dx11.\+cpp. There are 20+ examples in the \href{https://github.com/ocornut/imgui/blob/master/examples/}{\texttt{examples/\+}} folder. See \href{https://github.com/ocornut/imgui/blob/master/docs/EXAMPLES.md}{\texttt{EXAMPLES.\+MD}} for details.

\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md522}{}\doxysubsubsection{\texorpdfstring{Using a custom engine?}{Using a custom engine?}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md522}
You will likely be tempted to start by rewrite your own backend using your own custom/\+high-\/level facilities...~\newline
 Think twice!

TL;DR;
\begin{DoxyItemize}
\item Writing your own Renderer \doxylink{class_backend}{Backend} is easy.
\item Writing your own Platform \doxylink{class_backend}{Backend} is harder and you are more likely to introduce bugs.
\item {\bfseries{It is unlikely you will add value to your project by creating your own backend.}}
\end{DoxyItemize}

{\bfseries{Consider using the existing backends as-\/is}}. You will save lots of time integrating the library. Standard backends are battle-\/tested and handle subtleties that you are likely to implement incorrectly. You can LATER decide to rewrite yourself a custom backend if you really need to. In most situations, custom backends have fewer features and more bugs than the standard backends we provide. If you want portability, you can use multiple backends and choose between them either at compile time or at runtime.

{\bfseries{Example A}}:\+ your engine is built over Windows + Direct\+X11 but you have your own high-\/level rendering system layered over Direct\+X11.~\newline
 Suggestion:\+ try using imgui\+\_\+impl\+\_\+win32.\+cpp + imgui\+\_\+impl\+\_\+dx11.\+cpp first. Once it works, if you really need it, you can replace the imgui\+\_\+impl\+\_\+dx11.\+cpp code with a custom renderer using your own rendering functions, and keep using the standard Win32 code etc.

{\bfseries{Example B}}:\+ your engine runs on Windows, Mac, Linux and uses Direct\+X11, Metal, and Vulkan respectively.~\newline
 Suggestion:\+ use multiple generic backends! Once it works, if you really need it, you can replace parts of backends with your own abstractions.

{\bfseries{Example C}}:\+ your engine runs on platforms we can\textquotesingle{}t provide public backends for (e.\+g. PS4/\+\+PS5, Switch), and you have high-\/level systems everywhere.~\newline
 Suggestion:\+ try using a non-\/portable backend first (e.\+g. win32 + underlying graphics API) to get your desktop builds working first. This will get you running faster and get your acquainted with how Dear Im\+Gui works and is setup. You can then rewrite a custom backend using your own engine API...

Also:\+ The \href{https://github.com/ocornut/imgui/wiki/Multi-Viewports}{\texttt{multi-\/viewports feature}} of the \textquotesingle{}docking\textquotesingle{} branch allows Dear Im\+Gui windows to be seamlessly detached from the main application window. This is achieved using an extra layer to the Platform and Renderer backends, which allows Dear Im\+Gui to communicate platform-\/specific requests such as:\+ "{}create an additional OS window"{}, "{}create a render context"{}, "{}get the OS position of this window"{}, but some things are more difficult "{}find OS window under mouse position BUT with some windows marked as passthrough"{}. See \textquotesingle{}\doxylink{struct_im_gui_platform_i_o}{Im\+Gui\+Platform\+IO}\textquotesingle{} for details. Supporting the multi-\/viewports feature correctly using 100\% of your own abstractions is more difficult than supporting single-\/viewport. If you decide to use unmodified imgui\+\_\+impl\+\_\+\+XXXX.\+cpp files, you can automatically benefit from improvements and fixes related to viewports and platform windows without extra work on your side.\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md523}{}\doxysubsubsection{\texorpdfstring{Platform:\+ Implementing your Platform Backend}{Platform:\+ Implementing your Platform Backend}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md523}
The Platform backends in impl\+\_\+impl\+\_\+\+XXX.\+cpp files contain many implementations.

{\bfseries{In your {\ttfamily Im\+Gui\+\_\+\+Impl\+XXX\+\_\+\+Init()} function:\+}}
\begin{DoxyItemize}
\item You can allocate your backend data and use {\ttfamily io.\+Backend\+Platform\+User\+Data} to store/\+retrieve it later.
\item Set {\ttfamily io.\+Backend\+Platform\+Name} to a name {\ttfamily "{}imgui\+\_\+impl\+\_\+xxxx"{}} which will be available in e.\+g. About box.
\item Set {\ttfamily io.\+Backend\+Platform\+User\+Data} to your backend data.
\item Set {\ttfamily io.\+Backend\+Flags} with supported optional features:\+
\begin{DoxyItemize}
\item {\ttfamily Im\+Gui\+Backend\+Flags\+\_\+\+Has\+Gamepad}:\+ supports gamepad and currently has one connected.
\item {\ttfamily Im\+Gui\+Backend\+Flags\+\_\+\+Has\+Mouse\+Cursors}:\+ supports honoring Get\+Mouse\+Cursor() value to change the OS cursor shape.
\item {\ttfamily Im\+Gui\+Backend\+Flags\+\_\+\+Has\+Set\+Mouse\+Pos}:\+ supports io.\+Want\+Set\+Mouse\+Pos requests to reposition the OS mouse position (only used if io.\+Config\+Nav\+Move\+Set\+Mouse\+Pos is set).
\item {\ttfamily Im\+Gui\+Backend\+Flags\+\_\+\+Platform\+Has\+Viewports} supports multiple viewports. (multi-\/viewports only)
\item {\ttfamily Im\+Gui\+Backend\+Flags\+\_\+\+Has\+Mouse\+Hovered\+Viewport} supports calling io.\+Add\+Mouse\+Viewport\+Event() with the viewport under the mouse. IF POSSIBLE, ignore viewports with the Im\+Gui\+Viewport\+Flags\+\_\+\+No\+Inputs flag. If this cannot be done, Dear Im\+Gui needs to use a flawed heuristic to find the viewport under mouse position, as it doesn\textquotesingle{}t know about foreign windows. (multi-\/viewports only)
\end{DoxyItemize}
\end{DoxyItemize}

{\bfseries{In your {\ttfamily Im\+Gui\+\_\+\+Impl\+XXX\+\_\+\+New\+Frame()} function:\+}}
\begin{DoxyItemize}
\item Set {\ttfamily io.\+Delta\+Time} to the time elapsed (in seconds) since last frame.
\item Set {\ttfamily io.\+Display\+Size} to your window size.
\item Set {\ttfamily io.\+Display\+Frame\+Buffer\+Size} to your window pixel density (mac\+OS/\+i\+OS only).
\item Update mouse cursor shape is supported.
\end{DoxyItemize}

{\bfseries{In your {\ttfamily Im\+Gui\+\_\+\+Impl\+XXX\+\_\+\+New\+Frame()} function or event handlers:\+}}
\begin{DoxyItemize}
\item {\bfseries{Mouse Support}}
\begin{DoxyItemize}
\item Use {\ttfamily io.\+Add\+Mouse\+Pos\+Event()}, {\ttfamily io.\+Add\+Mouse\+Button\+Event()}, {\ttfamily io.\+Add\+Mouse\+Wheel\+Event()} to pass mouse events.
\item Use {\ttfamily io.\+Add\+Mouse\+Source\+Event()} if you are able to distinguish Mouse from Touch\+Screen from \doxylink{struct_pen}{Pen} inputs. Touch\+Screen and \doxylink{struct_pen}{Pen} inputs requires different logic for some Dear Im\+Gui features.
\item Use {\ttfamily io.\+Add\+Mouse\+Viewport\+Event()} to specify which viewport/\+\+OS window is being hovered by the mouse. Read instructions carefully as this is not as simple as it seems! (multi-\/viewports only)
\end{DoxyItemize}
\item {\bfseries{Keyboard Support}}
\begin{DoxyItemize}
\item Use {\ttfamily io.\+Add\+Key\+Event()} to pass key events.
\item Use {\ttfamily io.\+Add\+Input\+Character()} to pass text/\+character events.
\end{DoxyItemize}
\item {\bfseries{Gamepad Support}}
\begin{DoxyItemize}
\item Use {\ttfamily io.\+Add\+Key\+Event()} and {\ttfamily io.\+Add\+Key\+Analog\+Event()} to pass gamepad events, using {\ttfamily Im\+Gui\+Key\+\_\+\+Gamepad\+XXX} values.
\end{DoxyItemize}
\item {\bfseries{Miscellaneous}}
\begin{DoxyItemize}
\item Clipboard Support:\+ setup {\ttfamily Platform\+\_\+\+Get\+Clipboard\+Text\+Fn()}, {\ttfamily Platform\+\_\+\+Set\+Clipboard\+Text\+Fn()} handlers in {\ttfamily \doxylink{struct_im_gui_platform_i_o}{Im\+Gui\+Platform\+IO}}.
\item Open in Shell support:\+ setup {\ttfamily Platform\+\_\+\+Open\+In\+Shell\+Fn()} handler in {\ttfamily \doxylink{struct_im_gui_platform_i_o}{Im\+Gui\+Platform\+IO}}.
\item IME Support:\+ setup {\ttfamily Platform\+\_\+\+Set\+Ime\+Data\+Fn()} handler in {\ttfamily \doxylink{struct_im_gui_platform_i_o}{Im\+Gui\+Platform\+IO}}.
\item Use {\ttfamily io.\+Add\+Focus\+Event()} to notify when application window gets focused/\+unfocused.
\end{DoxyItemize}
\item {\bfseries{Multi-\/viewport Support}}
\begin{DoxyItemize}
\item Update monitor list if supported.
\item Setup all required handlers in {\ttfamily \doxylink{struct_im_gui_platform_i_o}{Im\+Gui\+Platform\+IO}} to create/\+destroy/\+move/\+resize/\+title/\+focus/\+etc. windows.
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md524}{}\doxysubsubsection{\texorpdfstring{Rendering:\+ Implementing your Render\+Draw\+Data function}{Rendering:\+ Implementing your Render\+Draw\+Data function}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md524}

Version \href{https://github.com/ocornut/imgui/releases/tag/v1.92.0}{\texttt{1.\+92.\+0}} (June 2025), added texture support in Rendering Backends, which is the backbone for supporting dynamic font scaling among other things.

{\bfseries{In order to move forward and take advantage of all new features, support for {\ttfamily Im\+Gui\+Backend\+Flags\+\_\+\+Renderer\+Has\+Textures} will likely be REQUIRED for all backends before June 2026.}}

{\ttfamily \doxylink{struct_im_font_atlas}{Im\+Font\+Atlas}} functions such as {\ttfamily Build()}, {\ttfamily Get\+Tex\+Data\+As\+RGBA32()}, {\ttfamily Get\+Tex\+Data\+As\+Alpha8()}, {\ttfamily Set\+Tex\+ID()}, {\ttfamily Is\+Built()} were obsoleted in favor if iterating a {\ttfamily Textures\+[]\+} array and updating their state when requested by Dear Im\+Gui.

{\bfseries{Instructions:\+}}


\begin{DoxyItemize}
\item Set {\ttfamily Im\+Gui\+Backend\+Flags\+\_\+\+Renderer\+Has\+Textures} to signify your backend can handle the feature.
\item During rendering, e.\+g. in your Render\+Draw\+Data function, iterate {\ttfamily Im\+Draw\+Data-\/\texorpdfstring{$>$}{>}Textures} array and process all textures.
\item During shutdown, iterate the {\ttfamily Im\+Gui::\+\+Get\+Platform\+IO().Textures} and destroy all textures.
\item (Both arrays are {\ttfamily \doxylink{struct_im_vector}{Im\+Vector}\texorpdfstring{$<$}{<}Im\+Texture\+Data\texorpdfstring{$\ast$}{*}\texorpdfstring{$>$}{>}}. They are only in different location because:\+ to allow advanced users to perform multi-\/threaded rendering, we store a pointer to the texture list in \doxylink{struct_im_draw_data}{Im\+Draw\+Data}, with the aim that multi-\/threaded rendering users replace it with their own pointer.)
\end{DoxyItemize}
