(As of October 2024, but things move quickly and we don\textquotesingle{}t update this document often.)

In modern times, all the browsers you probably care about (Chrome, Firefox, Edge, and Safari, on Windows, mac\+OS, Linux, i\+OS and Android), support some reasonable base configurations:\+


\begin{DoxyItemize}
\item Web\+Assembly (don\textquotesingle{}t bother with asm.\+js any more)
\item Web\+GL (which will look like Open\+GL ES 2 or 3 to your app).
\item Threads (see caveats, though!)
\item Game controllers
\item Autoupdating (so you can assume they have a recent version of the browser)
\end{DoxyItemize}

All this to say we\textquotesingle{}re at the point where you don\textquotesingle{}t have to make a lot of concessions to get even a fairly complex SDL-\/based game up and running.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1301}{}\doxysection{\texorpdfstring{RTFM}{RTFM}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1301}
This document is a quick rundown of some high-\/level details. The documentation at \href{https://emscripten.org/}{\texttt{emscripten.\+org}} is vast and extremely detailed for a wide variety of topics, and you should at least skim through it at some point.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1302}{}\doxysection{\texorpdfstring{Porting your app to Emscripten}{Porting your app to Emscripten}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1302}
Many many things just need some simple adjustments and they\textquotesingle{}ll compile like any other C/\+\+C++ code, as long as SDL was handling the platform-\/specific work for your program.

First:\+ assembly language code has to go. Replace it with C. You can even use \href{https://emscripten.org/docs/porting/simd.html}{\texttt{x86 SIMD intrinsic functions in Emscripten}}!

Second:\+ Middleware has to go. If you have a third-\/party library you link against, you either need an Emscripten port of it, or the source code to it to compile yourself, or you need to remove it.

Third:\+ If your program starts in a function called main(), you need to get out of it and into a function that gets called repeatedly, and returns quickly, called a mainloop.

Somewhere in your program, you probably have something that looks like a more complicated version of this:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{void}\ main(\textcolor{keywordtype}{void})}
\DoxyCodeLine{\{}
\DoxyCodeLine{\ \ \ \ initialize\_the\_game();}
\DoxyCodeLine{\ \ \ \ \textcolor{keywordflow}{while}\ (game\_is\_still\_running)\ \{}

\DoxyCodeLine{\ \ \ \ \ \ \ \ think\_about\_stuff();}

\DoxyCodeLine{\ \ \ \ \}}

\DoxyCodeLine{\}}

This will not work on Emscripten, because the main thread needs to be free to do stuff and can\textquotesingle{}t sit in this loop forever. So Emscripten lets you set up a \href{https://emscripten.org/docs/porting/emscripten-runtime-environment.html\#browser-main-loop}{\texttt{mainloop}}.

Basically, {\ttfamily emscripten\+\_\+set\+\_\+main\+\_\+loop(mainloop, 0, 1);} says "{}run \`{}mainloop\`{} over and over until I end the program."{} The function will run, and return, freeing the main thread for other tasks, and then run again when it\textquotesingle{}s time. The {\ttfamily 1} parameter does some magic to make your main() function end immediately; this is useful because you don\textquotesingle{}t want any shutdown code that might be sitting below this code to actually run if main() were to continue on, since we\textquotesingle{}re just getting started.

Another option is to use SDL\textquotesingle{}s main callbacks, which handle this for you without platform-\/specific code in your app. Please refer to \href{https://wiki.libsdl.org/SDL3/README-main-functions\#main-callbacks-in-sdl3}{\texttt{the wiki}} or {\ttfamily docs/\+\+README-\/main-\/functions.\+md} in the SDL source code.

There\textquotesingle{}s a lot of little details that are beyond the scope of this document, but that\textquotesingle{}s the biggest initial set of hurdles to porting your app to the web.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1303}{}\doxysection{\texorpdfstring{Do you need threads?}{Do you need threads?}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1303}
If you plan to use threads, they work on all major browsers now. HOWEVER, they bring with them a lot of careful considerations. Rendering {\itshape must} be done on the main thread. This is a general guideline for many platforms, but a hard requirement on the web.

Many other things also must happen on the main thread; often times SDL and Emscripten make efforts to "{}proxy"{} work to the main thread that must be there, but you have to be careful (and read more detailed documentation than this for the finer points).

Even when using threads, your main thread needs to set an Emscripten mainloop (or use SDL\textquotesingle{}s main callbacks) that runs quickly and returns, or things will fail to work correctly.

You should definitely read \href{https://emscripten.org/docs/porting/pthreads.html}{\texttt{Emscripten\textquotesingle{}s pthreads docs}} for all the finer points. Mostly SDL\textquotesingle{}s thread API will work as expected, but is built on pthreads, so it shares the same little incompatibilities that are documented there, such as where you can use a mutex, and when a thread will start running, etc.

IMPORTANT:\+ You have to decide to either build something that uses threads or something that doesn\textquotesingle{}t; you can\textquotesingle{}t have one build that works everywhere. This is an Emscripten (or maybe Web\+Assembly? Or just web browsers in general?) limitation. If you aren\textquotesingle{}t using threads, it\textquotesingle{}s easier to not enable them at all, at build time.

If you use threads, you {\itshape have to} run from a web server that has \href{https://web.dev/why-coop-coep/}{\texttt{COOP/\+\+COEP headers set correctly}} or your program will fail to start at all.

If building with threads, {\ttfamily \+\_\+\+\_\+\+EMSCRIPTEN\+\_\+\+PTHREADS\+\_\+\+\_\+} will be defined for checking with the C preprocessor, so you can build something different depending on what sort of build you\textquotesingle{}re compiling.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1304}{}\doxysection{\texorpdfstring{Audio}{Audio}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1304}
Audio works as expected at the API level, but not exactly like other platforms.

You\textquotesingle{}ll only see a single default audio device. Audio recording also works; if the browser pops up a prompt to ask for permission to access the microphone, the SDL\+\_\+\+Open\+Audio\+Device call will succeed and start producing silence at a regular interval. Once the user approves the request, real audio data will flow. If the user denies it, the app is not informed and will just continue to receive silence.

Modern web browsers will not permit web pages to produce sound before the user has interacted with them (clicked or tapped on them, usually); this is for several reasons, not the least of which being that no one likes when a random browser tab suddenly starts making noise and the user has to scramble to figure out which and silence it.

SDL will allow you to open the audio device for playback in this circumstance, and your audio streams will consume data, but SDL will throw the audio data away until the user interacts with the page. This helps apps that depend on the audio callback to make progress, and also keeps audio playback in sync once the app is finally allowed to make noise.

There are two reasonable ways to deal with the silence at the app level:\+ if you are writing some sort of media player thing, where the user expects there to be a volume control when you mouseover the canvas, just default that control to a muted state; if the user clicks on the control to unmute it, on this first click, adjust your app\textquotesingle{}s volume appropriately, and SDL will also start actually feeding the data to the browser. This allows the media to play at start, and the user can reasonably opt-\/in to listening.

Many games do not have this sort of UI. For these, your best bet might be to write a little Javascript that puts up a "{}\+Click here to play!"{} UI, and upon the user clicking, remove that UI and then call the Emscripten app\textquotesingle{}s main() function. As far as the application knows, audio was able to play as soon as the program started, and since this magic happens in a little Javascript, you don\textquotesingle{}t have to change your C/\+\+C++ code at all to make it happen.

Please see the discussion at \href{https://github.com/libsdl-org/SDL/issues/6385}{\texttt{https:\+/\+/\+github.\+com/\+libsdl-\/org/\+\+SDL/\+issues/\+6385}} for some Javascript code to steal for this approach.

But if a game can just do without audio until the user clicks on the page, it will still operate correctly, as if the page was merely muted before then.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1305}{}\doxysection{\texorpdfstring{Rendering}{Rendering}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1305}
If you use SDL\textquotesingle{}s 2D render API, it will use GLES2 internally, which Emscripten will turn into Web\+GL calls. You can also use Open\+GL ES 2 directly by creating a GL context and drawing into it.

If the browser (and hardware) support Web\+GL 2, you can create an Open\+GL ES 3 context.

Calling SDL\+\_\+\+Render\+Present (or SDL\+\_\+\+GL\+\_\+\+Swap\+Window) will not actually present anything on the screen until your return from your mainloop function.

Note that SDL attempts to default to vsync {\itshape off} on all platforms. You almost certainly do {\itshape not} want this in Emscripten, however, as it will affect the efficiency of the mainloop. If using Open\+GL directly, you should call SDL\+\_\+\+GL\+\_\+\+Set\+Swap\+Interval(1) sometime near startup; if using the 2D render API, either create the renderer with with the property SDL\+\_\+\+PROP\+\_\+\+RENDERER\+\_\+\+CREATE\+\_\+\+PRESENT\+\_\+\+VSYNC\+\_\+\+NUMBER set to 1, or call SDL\+\_\+\+Set\+Render\+VSync(renderer, 1). If you don\textquotesingle{}t explicitly set vsync, you\textquotesingle{}ll get a higher (but perhaps unstable) framerate, and use more power, but it will still work. Choosing a vsync of 1 will use request\+Animation\+Frame if possible.

If you\textquotesingle{}re using the SDL main callbacks, the mainloop defaults to using request\+Animation\+Frame (effectively vsync), because it calls emscripten\+\_\+set\+\_\+main\+\_\+loop() with a zero fps. This is almost certainly what you want to do! Do this even if you aren\textquotesingle{}t using the main callbacks! SDL will attempt to accommodate the app if it messes with vsync settings, or doesn\textquotesingle{}t use request\+Animation\+Frame, but modern thinking is that this is the most efficient, consistent, and correct way to run a game in a web browser.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1306}{}\doxysection{\texorpdfstring{Building SDL/\+emscripten}{Building SDL/\+emscripten}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1306}
Use the latest stable Emscripten release!

It\textquotesingle{}s possible to build SDL with older Emscripten releases, such as 3.\+x, but several things will be silently broken, as bugs got fixed and web standards solidified over time. At the time of this writing, Emscripten 4.\+0.\+x is the current stable release. You\textquotesingle{}re encouraged to install the latest stable release ({\ttfamily emsdk install latest ; emsdk activate latest} if using Emscripten\textquotesingle{}s setup script), and make sure you\textquotesingle{}re reasonably up to date as time goes on.

Build:\+

This works on Linux/\+\+Unix and mac\+OS. Please send comments about Windows.

Make sure you\textquotesingle{}ve \href{https://emscripten.org/docs/getting_started/downloads.html}{\texttt{installed emsdk}} first, and run {\ttfamily source emsdk\+\_\+env.\+sh} at the command line so it finds the tools.

(These cmake options might be overkill, but this has worked for me.)

\DoxyCodeLine{cd\ build}
\DoxyCodeLine{emcmake\ cmake\ ..}
\DoxyCodeLine{\#\ you\ can\ also\ try\ \`{}emcmake\ cmake\ -\/G\ Ninja\ ..`\ and\ then\ use\ \`{}ninja`\ instead\ of\ this\ command.}
\DoxyCodeLine{emmake\ make\ -\/j4}

\end{DoxyCode}


If you want to build with thread support, something like this works:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{mkdir\ build}
\DoxyCodeLine{cd\ build}
\DoxyCodeLine{emcmake\ cmake\ -\/DSDL\_PTHREADS=ON\ ..}

\end{DoxyCode}


To build the tests, add {\ttfamily -\/DSDL\+\_\+\+TESTS=ON} to the {\ttfamily emcmake cmake} command line. To build the examples, add {\ttfamily -\/DSDL\+\_\+\+EXAMPLES=ON} to the {\ttfamily emcmake cmake} command line.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1307}{}\doxysection{\texorpdfstring{Building your app}{Building your app}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1307}
You need to compile with {\ttfamily emcc} instead of {\ttfamily gcc} or {\ttfamily clang} or whatever, but mostly it uses the same command line arguments as Clang.

Link against the lib\+SDL3.\+a file you generated by building SDL.

Usually you would produce a binary like this:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{gcc\ -\/o\ mygame\ mygame.c\ \ \#\ or\ whatever}

\end{DoxyCode}


But for Emscripten, you want to output something else:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{emcc\ -\/o\ index.html\ mygame.c}

\end{DoxyCode}


This will produce several files...support Javascript and Web\+Assembly (.wasm) files. The {\ttfamily -\/o index.\+html} will produce a simple HTML page that loads and runs your app. You will (probably) eventually want to replace or customize that file and do {\ttfamily -\/o index.\+js} instead to just build the code pieces.

If you\textquotesingle{}re working on a program of any serious size, you\textquotesingle{}ll likely need to link with {\ttfamily -\/s ALLOW\+\_\+\+MEMORY\+\_\+\+GROWTH=1 -\/s MAXIMUM\+\_\+\+MEMORY=1gb} to get access to more memory. If using pthreads, you\textquotesingle{}ll need the {\ttfamily -\/s MAXIMUM\+\_\+\+MEMORY=1gb} or the app will fail to start on i\+OS browsers, but this might be a bug that goes away in the future.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1308}{}\doxysection{\texorpdfstring{Data files}{Data files}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1308}
Your game probably has data files. Here\textquotesingle{}s how to access them.

Filesystem access works like a Unix filesystem; you have a single directory tree, possibly interpolated from several mounted locations, no drive letters, \textquotesingle{}/\+\textquotesingle{} for a path separator. You can access them with standard file APIs like open() or fopen() or \doxylink{struct_s_d_l___i_o_stream}{SDL\+\_\+\+IOStream}. You can read or write from the filesystem.

By default, you probably have a "{}\+MEMFS"{} filesystem (all files are stored in memory, but access to them is immediate and doesn\textquotesingle{}t need to block). There are other options, like "{}\+IDBFS"{} (files are stored in a local database, so they don\textquotesingle{}t need to be in RAM all the time and they can persist between runs of the program, but access is not synchronous). You can mix and match these file systems, mounting a MEMFS filesystem at one place and idbfs elsewhere, etc, but that\textquotesingle{}s beyond the scope of this document. Please refer to Emscripten\textquotesingle{}s \href{https://emscripten.org/docs/porting/files/file_systems_overview.html}{\texttt{page on the topic}} for more info.

The {\itshape easiest} (but not the best) way to get at your data files is to embed them in the app itself. Emscripten\textquotesingle{}s linker has support for automating this.


\begin{DoxyCode}{0}
\DoxyCodeLine{emcc\ -\/o\ index.html\ loopwave.c\ -\/-\/embed-\/file\ ../test/sample.wav@/sounds/sample.wav}

\end{DoxyCode}


This will pack ../\+test/\+sample.wav in your app, and make it available at "{}/\+sounds/\+sample.\+wav"{} at runtime. Emscripten makes sure this data is available before your main() function runs, and since it\textquotesingle{}s in MEMFS, you can just read it like you do on other platforms. {\ttfamily -\/-\/embed-\/file} can also accept a directory to pack an entire tree, and you can specify the argument multiple times to pack unrelated things into the final installation.

Note that this is absolutely the best approach if you have a few small files to include and shouldn\textquotesingle{}t worry about the issue further. However, if you have hundreds of megabytes and/\+or thousands of files, this is not so great, since the user will download it all every time they load your page, and it all has to live in memory at runtime.

\href{https://emscripten.org/docs/porting/files/packaging_files.html}{\texttt{Emscripten\textquotesingle{}s documentation on the matter}} gives other options and details, and is worth a read.

Please also read the next section on persistent storage, for a little help from SDL.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1309}{}\doxysection{\texorpdfstring{Automount persistent storage}{Automount persistent storage}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1309}
The file tree in Emscripten is provided by MEMFS by default, which stores all files in RAM. This is often what you want, because it\textquotesingle{}s fast and can be accessed with the usual synchronous i/\+o functions like fopen or SDL\+\_\+\+IOFrom\+File. You can also write files to MEMFS, but when the browser tab goes away, so do the files. But we want things like high scores, save games, etc, to still exist if we reload the game later.

For this, Emscripten offers IDBFS, which backs files with the browser\textquotesingle{}s \href{https://en.wikipedia.org/wiki/IndexedDB}{\texttt{Indexed\+DB}} functionality.

To use this, the app has to mount the IDBFS filesystem somewhere in the virtual file tree, and then wait for it to sync up. This needs to be done in Javascript code. The sync will not complete until at least one (but possibly several) iterations of the mainloop have passed, which means you can not access any saved files during main() or SDL\+\_\+\+App\+Init() by default.

SDL can solve this problem for you:\+ it can be built to automatically mount the persistent files from IDBFS to a specific place in the file tree and wait until the sync has completed before calling main() or SDL\+\_\+\+App\+Init(), so to your C code, it looks like the files were always available.

To use this functionality, set the CMake variable {\ttfamily SDL\+\_\+\+EMSCRIPTEN\+\_\+\+PERSISTENT\+\_\+\+PATH} to a path in the filetree where persistent storage should be mounted:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{mkdir\ build}
\DoxyCodeLine{cd\ build}
\DoxyCodeLine{emcmake\ cmake\ -\/DSDL\_EMSCRIPTEN\_PERSISTENT\_PATH=/storage\ ..}

\end{DoxyCode}


You should also link your app with {\ttfamily -\/lidbfs.\+js}. If your project links to SDL using CMake\textquotesingle{}s find\+\_\+package(\+SDL3), or uses {\ttfamily pkg-\/config sdl3 -\/-\/libs}, this will be handled for you when used with an SDL built with {\ttfamily -\/DSDL\+\_\+\+EMSCRIPTEN\+\_\+\+PERSISTENT\+\_\+\+PATH}.

Now {\ttfamily /\+storage} will be prepared when your program runs, and SDL\+\_\+\+Get\+Pref\+Path() will return a directory under that path. The storage is mounted with the {\ttfamily auto\+Persist:\+ true} option, so when you write to that tree, whether with SDL APIs or other functions like fopen(), Emscripten will know it needs to sync that data back to the persistent database, and will do so automatically within the next few iterations of the mainloop.

It\textquotesingle{}s best to assume the sync will take a few frames to complete, and the data is not safe until it does.

To summarize how to automate this:\+


\begin{DoxyItemize}
\item Build with {\ttfamily emcmake cmake -\/DSDL\+\_\+\+EMSCRIPTEN\+\_\+\+PERSISTENT\+\_\+\+PATH=/\+storage}
\item Link your app with {\ttfamily -\/lidbfs.\+js} if not handled automatically.
\item Write under {\ttfamily /\+storage}, or use SDL\+\_\+\+Get\+Pref\+Path()
\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1310}{}\doxysection{\texorpdfstring{Customizing index.\+html}{Customizing index.\+html}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1310}
You don\textquotesingle{}t have to use the HTML that Emscripten produces; the above examples use {\ttfamily emcc -\/o index.\+html}, but you can {\ttfamily -\/o index.\+js} instead to just output code without an HTML page, and then provide your own. This is desirable for shipping products, even though the Emscripten-\/provided HTML is fine for prototyping. Certain things {\itshape must} be in the HTML file or your program will not function correctly (or function at all). The specifics are beyond the scope of this document, but it\textquotesingle{}s likely best to start with the Emscripten HTML and customize it, instead of starting from scratch.

The {\ttfamily \texorpdfstring{$<$}{<}canvas\texorpdfstring{$>$}{>}} element in the HTML {\itshape must not} have a border or padding, or things will break in unexpected ways. This can be surprising when customizing the page\textquotesingle{}s look. Plan accordingly.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1311}{}\doxysection{\texorpdfstring{Debugging}{Debugging}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1311}
Debugging web apps is a mixed bag. You should compile and link with {\ttfamily -\/gsource-\/map}, which embeds a ton of source-\/level debugging information into the build, and make sure {\itshape the app source code is available on the web server}, which is often a scary proposition for various reasons.

When you debug from the browser\textquotesingle{}s tools and hit a breakpoint, you can step through the actual C/\+\+C++ source code, though, which can be nice.

If you try debugging in Firefox and it doesn\textquotesingle{}t work well for no apparent reason, try Chrome, and vice-\/versa. These tools are still relatively new, and improving all the time.

SDL\+\_\+\+Log() (or printf) will write to the Javascript console, so printf-\/style debugging can be easier than setting up a build for proper debugging. Use whatever tools work best for you.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1312}{}\doxysection{\texorpdfstring{Questions?}{Questions?}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-emscripten_autotoc_md1312}
