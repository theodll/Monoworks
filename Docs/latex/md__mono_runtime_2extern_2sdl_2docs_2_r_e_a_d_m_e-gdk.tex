\chapter{GDK }
\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk}{}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk}\index{GDK@{GDK}}
This port allows SDL applications to run via Microsoft\textquotesingle{}s Game Development Kit (GDK).

Windows (GDK) and Xbox One/\+\+Xbox Series (GDKX) are both supported and all the required code is included in this public SDL release. However, only licensed Xbox developers have access to the GDKX libraries which will allow you to build the Xbox targets.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1313}{}\doxysection{\texorpdfstring{Requirements}{Requirements}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1313}

\begin{DoxyItemize}
\item Microsoft Visual Studio 2022 (in theory, it should also work in 2017 or 2019, but this has not been tested)
\item Microsoft GDK October 2023 Update 1 or newer (public release \href{https://github.com/microsoft/GDK/releases/tag/October_2023_Update_1}{\texttt{here}})
\item For Xbox, you will need the corresponding GDKX version (licensed developers only)
\item To publish a package or successfully authenticate a user, you will need to create an app id/\+configure services in Partner Center. However, for local testing purposes (without authenticating on Xbox Live), the test identifiers used by the GDK test programs in the included solution work.
\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1314}{}\doxysection{\texorpdfstring{Windows GDK Status}{Windows GDK Status}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1314}
The Windows GDK port supports the full set of Win32 APIs, renderers, controllers, input devices, etc., as the normal Windows x64 build of SDL.


\begin{DoxyItemize}
\item Additionally, the GDK port adds the following:\+
\begin{DoxyItemize}
\item Compile-\/time platform detection for SDL programs. The {\ttfamily SDL\+\_\+\+PLATFORM\+\_\+\+GDK} is {\ttfamily \#define}d on every GDK platform, and the {\ttfamily SDL\+\_\+\+PLATFORM\+\_\+\+WINGDK} is {\ttfamily \#define}d on Windows GDK, specifically. (This distinction exists because other GDK platforms support a smaller subset of functionality. This allows you to mark code for "{}any"{} GDK separate from Windows GDK.)
\item GDK-\/specific setup:\+
\begin{DoxyItemize}
\item Initializing/\+uninitializing the game runtime, and initializing Xbox Live services
\item Creating a global task queue and setting it as the default for the process. When running any async operations, passing in {\ttfamily NULL} as the task queue will make the task get added to the global task queue.
\end{DoxyItemize}
\item An implementation on {\ttfamily Win\+Main} that performs the above GDK setup that you can use by \+\#include\textquotesingle{}ing \doxylink{_s_d_l__main_8h_source}{SDL\+\_\+main.\+h} in the source file that includes your standard main() function. If you are unable to do this, you can instead manually call {\ttfamily SDL\+\_\+\+Run\+App} from your entry point, passing in your {\ttfamily SDL\+\_\+main} function and {\ttfamily NULL} as the parameters. To use {\ttfamily SDL\+\_\+\+Run\+App}, {\ttfamily \#define SDL\+\_\+\+MAIN\+\_\+\+HANDLED} before {\ttfamily \#include \texorpdfstring{$<$}{<}\doxylink{_s_d_l__main_8h_source}{SDL3/\+\+SDL\+\_\+main.\+h}\texorpdfstring{$>$}{>}}.
\item Global task queue callbacks are dispatched during {\ttfamily SDL\+\_\+\+Pump\+Events} (which is also called internally if using {\ttfamily SDL\+\_\+\+Poll\+Event}).
\item You can get the handle of the global task queue through {\ttfamily SDL\+\_\+\+Get\+GDKTask\+Queue}, if needed. When done with the queue, be sure to use {\ttfamily XTask\+Queue\+Close\+Handle} to decrement the reference count (otherwise it will cause a resource leak).
\end{DoxyItemize}
\item Single-\/player games have some additional features available:\+
\begin{DoxyItemize}
\item Call {\ttfamily SDL\+\_\+\+Get\+GDKDefault\+User} to get the default XUser\+Handle pointer.
\item {\ttfamily SDL\+\_\+\+Get\+Pref\+Path} and {\ttfamily \doxylink{struct_s_d_l___storage}{SDL\+\_\+\+Storage}} still work, but only for single-\/player titles.
\begin{DoxyItemize}
\item To enable support for these APIs, use {\ttfamily SDL\+\_\+\+Set\+Hint} to set {\ttfamily SDL\+\_\+\+GDK\+\_\+\+SERVICE\+\_\+\+CONFIGURATION\+\_\+\+ID} to your SCID string at startup (it should be formatted like {\ttfamily "{}00000000-\/0000-\/0000-\/0000-\/000000000000"{}})
\end{DoxyItemize}
\end{DoxyItemize}

These functions mostly wrap around async APIs, and thus should be treated as synchronous alternatives. Also note that the single-\/player functions return on any OS errors, so be sure to validate the return values!
\item What doesn\textquotesingle{}t work:\+
\begin{DoxyItemize}
\item Compilation with anything other than through the included Visual C++ solution file
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1315}{}\doxysection{\texorpdfstring{Visual\+C-\/\+GDK Solution}{Visual\+C-\/\+GDK Solution}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1315}
The included {\ttfamily Visual\+C-\/\+GDK/\+\+SDL.\+sln} solution includes the following targets for the Gaming.\+Desktop.\+x64 configuration:\+


\begin{DoxyItemize}
\item SDL3 (DLL) -\/ This is the typical SDL3.\+dll, but for Gaming.\+Desktop.\+x64.
\item tests/\+testgamecontroller -\/ Standard SDL test program demonstrating controller functionality.
\item tests/\+testgdk -\/ GDK-\/specific test program that demonstrates using the global task queue to login a user into Xbox Live. {\itshape NOTE}:\+ As of the June 2022 GDK, you cannot test user logins without a valid Title ID and MSAApp\+Id. You will need to manually change the identifiers in the {\ttfamily Microsoft\+Game.\+config} to your valid IDs from Partner Center if you wish to test this.
\item tests/\+testsprite -\/ Standard SDL test program demonstrating sprite drawing functionality.
\end{DoxyItemize}

If you set one of the test programs as a startup project, you can run it directly from Visual Studio.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1316}{}\doxysection{\texorpdfstring{Windows GDK Setup, Detailed Steps}{Windows GDK Setup, Detailed Steps}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1316}
These steps assume you already have a game using SDL that runs on Windows x64 along with a corresponding Visual Studio solution file for the x64 version. If you don\textquotesingle{}t have this, it\textquotesingle{}s easiest to use one of the test program vcxproj files in the {\ttfamily Visual\+C-\/\+GDK} directory as a starting point, though you will still need to do most of the steps below.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1317}{}\doxysubsection{\texorpdfstring{1. Add a Gaming.\+Desktop.\+x64 Configuration}{1. Add a Gaming.\+Desktop.\+x64 Configuration}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1317}
In your game\textquotesingle{}s existing Visual Studio Solution, go to Build \texorpdfstring{$>$}{>} Configuration Manager. From the "{}\+Active solution platform"{} drop-\/down select "{}\+New..."{}. From the drop-\/down list, select Gaming.\+Desktop.\+x64 and copy the settings from the x64 configuration.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1318}{}\doxysubsection{\texorpdfstring{2. Build SDL3 for GDK}{2. Build SDL3 for GDK}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1318}
Open {\ttfamily Visual\+C-\/\+GDK/\+\+SDL.\+sln} in Visual Studio, you need to build the SDL3 target for the Gaming.\+Desktop.\+x64 platform (Release is recommended). You will need to copy/\+keep track of the {\ttfamily SDL3.\+dll}, {\ttfamily XCurl.\+dll} (which is output by Gaming.\+Desktop.\+x64), and {\ttfamily SDL3.\+lib} output files for your game project.

{\itshape Alternatively}, you could setup your solution file to instead reference the SDL3 project file targets from the SDL source, and add those projects as a dependency. This would mean that SDL3 would be built when your game is built.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1319}{}\doxysubsection{\texorpdfstring{3. Configuring Project Settings}{3. Configuring Project Settings}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1319}
While the Gaming.\+Desktop.\+x64 configuration sets most of the required settings, there are some additional items to configure for your game project under the Gaming.\+Desktop.\+x64 Configuration:\+


\begin{DoxyItemize}
\item Under C/\+\+C++ \texorpdfstring{$>$}{>} General \texorpdfstring{$>$}{>} Additional Include Directories, make sure the {\ttfamily SDL/\+include} path is referenced
\item Under Linker \texorpdfstring{$>$}{>} General \texorpdfstring{$>$}{>} Additional Library Directories, make sure to reference the path where the newly-\/built SDL3.\+lib are
\item Under Linker \texorpdfstring{$>$}{>} Input \texorpdfstring{$>$}{>} Additional Dependencies, you need the following:\+
\begin{DoxyItemize}
\item {\ttfamily SDL3.\+lib}
\item {\ttfamily xgameruntime.\+lib}
\item {\ttfamily ../\+\+Microsoft.Xbox.\+Services.\+GDK.\+C.\+Thunks.\+lib}
\end{DoxyItemize}
\item Note that in general, the GDK libraries depend on the MSVC C/\+\+C++ runtime, so there is no way to remove this dependency from a GDK program that links against GDK.
\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1320}{}\doxysubsection{\texorpdfstring{4. Setting up SDL\+\_\+main}{4. Setting up SDL\+\_\+main}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1320}
Rather than using your own implementation of {\ttfamily Win\+Main}, it\textquotesingle{}s recommended that you instead {\ttfamily \#include \texorpdfstring{$<$}{<}\doxylink{_s_d_l__main_8h_source}{SDL3/\+\+SDL\+\_\+main.\+h}\texorpdfstring{$>$}{>}} and declare a standard main function. If you are unable to do this, you can instead manually call {\ttfamily SDL\+\_\+\+Run\+App} from your entry point, passing in your {\ttfamily SDL\+\_\+main} function and {\ttfamily NULL} as the parameters; in that case {\ttfamily \#define SDL\+\_\+\+MAIN\+\_\+\+HANDLED} before including \doxylink{_s_d_l__main_8h_source}{SDL\+\_\+main.\+h}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1321}{}\doxysubsection{\texorpdfstring{5. Required DLLs}{5. Required DLLs}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1321}
The game will not launch in the debugger unless required DLLs are included in the directory that contains the game\textquotesingle{}s .exe file. You need to make sure that the following files are copied into the directory:\+


\begin{DoxyItemize}
\item Your SDL3.\+dll
\item "{}\$(\+Console\+\_\+\+GRDKExt\+Lib\+Root)\+Xbox.\+Services.\+API.\+C\textbackslash{}\+Design\+Time\textbackslash{}\+Common\+Configuration\textbackslash{}\+Neutral\textbackslash{}\+Lib\textbackslash{}\+Release\textbackslash{}\+Microsoft.\+Xbox.\+Services.\+GDK.\+C.\+Thunks.\+dll"{}
\item XCurl.\+dll
\end{DoxyItemize}

You can either copy these in a post-\/build step, or you can add the dlls into the project and set its Configuration Properties \texorpdfstring{$>$}{>} General \texorpdfstring{$>$}{>} Item type to "{}\+Copy file,"{} which will also copy them into the output directory.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1322}{}\doxysubsection{\texorpdfstring{6. Setting up Microsoft\+Game.\+config}{6. Setting up Microsoft\+Game.\+config}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1322}
You can copy {\ttfamily Visual\+C-\/\+GDK/\+tests/\+testgdk/\+\+Microsoft\+Game.\+config} and use that as a starting point in your project. Minimally, you will want to change the Executable Name attribute, the Default\+Display\+Name, and the Description.

This file must be copied into the same directory as the game\textquotesingle{}s .exe file. As with the DLLs, you can either use a post-\/build step or the "{}\+Copy file"{} item type.

For basic testing, you do not need to change anything else in {\ttfamily Microsoft\+Game.\+config}. However, if you want to test any Xbox Live services (such as logging in users) {\itshape or} publish a package, you will need to setup a Game app on Partner Center.

Then, you need to set the following values to the values from Partner Center:\+


\begin{DoxyItemize}
\item Identity tag -\/ Name and Publisher attributes
\item Title\+Id
\item MSAApp\+Id
\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1323}{}\doxysubsection{\texorpdfstring{7. Adding Required Logos}{7. Adding Required Logos}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1323}
Several logo PNG files are required to be able to launch the game, even from the debugger. You can use the sample logos provided in {\ttfamily Visual\+C-\/\+GDK/\+logos}. As with the other files, they must be copied into the same directory as the game\textquotesingle{}s .exe file.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1324}{}\doxysubsection{\texorpdfstring{8. Copying any Data Files}{8. Copying any Data Files}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1324}
When debugging GDK games, there is no way to specify a working directory. Therefore, any required game data must also be copied into the output directory, likely in a post-\/build step.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1325}{}\doxysubsection{\texorpdfstring{9. Build and Run from Visual Studio}{9. Build and Run from Visual Studio}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1325}
At this point, you should be able to build and run your game from the Visual Studio Debugger. If you get any linker errors, make sure you double-\/check that you referenced all the required libs.

If you are testing Xbox Live functionality, it\textquotesingle{}s likely you will need to change to the Sandbox for your title. To do this:\+


\begin{DoxyEnumerate}
\item Run "{}\+Desktop VS 2022 Gaming Command Prompt"{} from the Start Menu
\item Switch the sandbox name with:\+ {\ttfamily Xbl\+PCSandbox SANDBOX.\#}
\item (To switch back to the retail sandbox):\+ {\ttfamily Xbl\+PCSandbox RETAIL}
\end{DoxyEnumerate}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1326}{}\doxysubsection{\texorpdfstring{10. Packaging and Installing Locally}{10. Packaging and Installing Locally}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1326}
You can use one of the test program\textquotesingle{}s {\ttfamily Package\+Layout.\+xml} as a starting point. Minimally, you will need to change the exe to the correct name and also reference any required game data. As with the other data files, it\textquotesingle{}s easiest if you have this copy to the output directory, although it\textquotesingle{}s not a requirement as you can specify relative paths to files.

To create the package:\+


\begin{DoxyEnumerate}
\item Run "{}\+Desktop VS 2022 Gaming Command Prompt"{} from the Start Menu
\item {\ttfamily cd} to the directory containing the {\ttfamily Package\+Layout.\+xml} with the correct paths (if you use the local path as in the sample package layout, this would be from your .exe output directory)
\item {\ttfamily mkdir Package} to create an output directory
\item To package the file into the {\ttfamily Package} directory, use:\+ {\ttfamily makepkg pack /\+f Package\+Layout.\+xml /\+lt /\+d . /\+nogameos /\+pc /\+pd Package}
\item To install the package, use:\+ {\ttfamily wdapp install PACKAGENAME.\+msixvc}
\item Once the package is installed, you can run it from the start menu.
\item As with when running from Visual Studio, if you need to test any Xbox Live functionality you must switch to the correct sandbox.
\end{DoxyEnumerate}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1327}{}\doxysection{\texorpdfstring{Xbox GDKX Setup}{Xbox GDKX Setup}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1327}
In general, the same process in the Windows GDK instructions work. There are just a few additional notes:\+
\begin{DoxyItemize}
\item For Xbox One consoles, use the Gaming.\+Xbox.\+Xbox\+One.\+x64 target
\item For Xbox Series consoles, use the Gaming.\+Xbox.\+Scarlett.\+x64 target
\item The Xbox One target sets the {\ttfamily SDL\+\_\+\+PLATFORM\+\_\+\+XBOXONE} define and the Xbox Series target sets the {\ttfamily SDL\+\_\+\+PLATFORM\+\_\+\+XBOXSERIES} define
\item You don\textquotesingle{}t need to link against the Xbox.\+Services Thunks lib nor include that dll in your package (it doesn\textquotesingle{}t exist for Xbox)
\item The shader blobs for Xbox are created in a pre-\/build step for the Xbox targets, rather than included in the source (due to NDA and version compatibility reasons)
\item To create a package, use:\+ {\ttfamily makepkg pack /\+f Package\+Layout.\+xml /\+lt /\+d . /\+pd Package}
\item To install the package, use:\+ {\ttfamily xbapp install \+[PACKAGE]\+.xvc}
\item For some reason, if you make changes that require SDL3.\+dll to build, and you are running through the debugger (instead of a package), you have to rebuild your .exe target for the debugger to recognize the dll has changed and needs to be transferred to the console again
\item While there are successful releases of Xbox titles using this port, it is not as extensively tested as other targets
\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1328}{}\doxysection{\texorpdfstring{Troubleshooting}{Troubleshooting}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1328}
\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1329}{}\doxysubsubsection{\texorpdfstring{Xbox Live Login does not work}{Xbox Live Login does not work}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1329}
As of June 2022 GDK, you must have a valid Title Id and MSAApp\+Id in order to test Xbox Live functionality such as user login. Make sure these are set correctly in the {\ttfamily Microsoft\+Game.\+config}. This means that even testgdk will not let you login without setting these properties to valid values.

Furthermore, confirm that your PC is set to the correct sandbox.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1330}{}\doxysubsubsection{\texorpdfstring{"{}\+The current user has already installed an unpackaged version of this app. A packaged version cannot replace this."{} error when installing}{"{}\+The current user has already installed an unpackaged version of this app. A packaged version cannot replace this."{} error when installing}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-gdk_autotoc_md1330}
Prior to June 2022 GDK, running from the Visual Studio debugger would still locally register the app (and it would appear on the start menu). To fix this, you have to uninstall it (it\textquotesingle{}s simplest to right click on it from the start menu to uninstall it).
