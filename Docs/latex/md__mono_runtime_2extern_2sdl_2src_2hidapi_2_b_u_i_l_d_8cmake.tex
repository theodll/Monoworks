To build HIDAPI with CMake, it has to be installedinstalled/\+available in the system.

Make sure you\textquotesingle{}ve checked \href{BUILD.md\#prerequisites}{\texttt{prerequisites}} and installed all required dependencies.

HIDAPI CMake build system allows you to build HIDAPI in two generally different ways:\+ 1) As a standalone package/\+librarystandalone package/\+library; 2) As part of a larger CMake projectpart of a larger CMake project.

{\bfseries{TL;DR}}:\+ if you\textquotesingle{}re experienced developer and have been working with CMake projects or have been written some of your own -\/ most of this document may not be of interest for you; just check variables names, its default values and the target names.\hypertarget{md__mono_runtime_2extern_2sdl_2src_2hidapi_2_b_u_i_l_d_8cmake_autotoc_md1636}{}\doxysection{\texorpdfstring{Installing CMake}{Installing CMake}}\label{md__mono_runtime_2extern_2sdl_2src_2hidapi_2_b_u_i_l_d_8cmake_autotoc_md1636}
CMake can be installed either using your system\textquotesingle{}s package manager, or by downloading an installer/\+prebuilt version from the \href{https://cmake.org/download/}{\texttt{official website}}.

On most \texorpdfstring{$\ast$}{*}nix systems, the preferred way to install CMake is via package manager, e.\+g. {\ttfamily sudo apt install cmake}.

On Windows CMake could be provided by your development environment (e.\+g. by Visual Studio Installer or Min\+GW installer), or you may install it system-\/wise using the installer from the official website.

On mac\+OS CMake may be installed by Homebrew/\+\+Mac\+Ports or using the installer from the official website.\hypertarget{md__mono_runtime_2extern_2sdl_2src_2hidapi_2_b_u_i_l_d_8cmake_autotoc_md1637}{}\doxysection{\texorpdfstring{Standalone package build}{Standalone package build}}\label{md__mono_runtime_2extern_2sdl_2src_2hidapi_2_b_u_i_l_d_8cmake_autotoc_md1637}

If HIDAPI isn\textquotesingle{}t installed in your system, or {\ttfamily find\+\_\+package} cannot find HIDAPI by default for any other reasons, the recommended way manually specify which HIDAPI package to use is via {\ttfamily hidapi\+\_\+\+ROOT} CMake variable, e.\+g.:\+ {\ttfamily -\/Dhidapi\+\_\+\+ROOT=\texorpdfstring{$<$}{<}path to HIDAPI installation prefix\texorpdfstring{$>$}{>}}.

Available CMake targets after successful {\ttfamily find\+\_\+package(hidapi)}:\+
\begin{DoxyItemize}
\item {\ttfamily hidapi::\+hidapi} -\/ indented to be used in most cases;
\item {\ttfamily hidapi::\+include} -\/ if you need only to include {\ttfamily \texorpdfstring{$<$}{<}\doxylink{hidapi_8h}{hidapi.\+h}\texorpdfstring{$>$}{>}} but not link against the library;

\item {\ttfamily hidapi::\+hidraw} -\/ available when hidraw backend is used/\+available on Linux;

There are several important differences in the behavior of HIDAPI CMake build system when CMake is built as standalone package vs subdirectory build:\+

1) In {\itshape standalone build} a number of standard and HIDAPI-\/specific variables are marked as {\itshape cache variables} or {\itshape options}. This is done for convenience:\+ when you\textquotesingle{}re building HIDAPI as a standalone package and using tools like {\ttfamily cmake-\/gui} -\/ those are highlighted as variables that can be changed and has some short description/\+documentation. E.\+g.:\+ ~\newline
 E.\+g.\+2:\+~\newline
 ~\newline
 When HIDAPI is built as a {\itshape subdirectory} -\/ {\bfseries{{\itshape none of the variables are marked for cache or as options}}} by HIDAPI. This is done to let the host project\textquotesingle{}s developer decide what is important (what needs to be highlighted) and what\textquotesingle{}s not.

2) The default behavior/\+default value for some of the variables is a bit different:\+
\begin{DoxyItemize}
\item by default, none of HIDAPI targets are \href{https://cmake.org/cmake/help/latest/command/install.html}{\texttt{installed}}; if required, HIDAPI targets can be installed by host project {\itshape after} including HIDAPI subdirectory (requires CMake 3.\+13 or later); {\bfseries{or}}, the default installation can be enabled by setting {\ttfamily HIDAPI\+\_\+\+INSTALL\+\_\+\+TARGETS} variable {\itshape before} including HIDAPI subdirectory. HIDAPI uses \href{https://cmake.org/cmake/help/latest/module/GNUInstallDirs.html}{\texttt{GNUInstall\+Dirs}} to specify install locations. Variables like {\ttfamily CMAKE\+\_\+\+INSTALL\+\_\+\+LIBDIR} can be used to control HIDAPI\textquotesingle{}s installation locations. E.\+g.:\+ {\ttfamily cmake @section autotoc\+\_\+md1641 enable the installation if you need it set(\+HIDAPI\+\_\+\+INSTALL\+\_\+\+TARGETS ON) @section autotoc\+\_\+md1642 (optionally) change default installation locations if it makes sense for your target platform, etc. set(CMAKE\+\_\+\+INSTALL\+\_\+\+LIBDIR "{}lib64"{}) add\+\_\+subdirectory(hidapi) }
\item HIDAPI prints its version during the configuration when built as a standalone package; to enable this for subdirectory builds -\/ set {\ttfamily HIDAPI\+\_\+\+PRINT\+\_\+\+VERSION} to TRUE before including HIDAPI;
\end{DoxyItemize}

3) In a subdirectory build, HIDAPI {\itshape doesn\textquotesingle{}t modify or set any of the CMake variables} that may change the build behavior. For instance, in a {\itshape standalone build}, if CMAKE\+\_\+\+BUILD\+\_\+\+TYPE or BUILD\+\_\+\+SHARED\+\_\+\+LIBS variables are not set, those are defaulted to "{}\+Release"{} and "{}\+TRUE"{} explicitly. In a {\itshape subdirectory build}, even if not set, those variables remain unchanged, so a host project\textquotesingle{}s developer has a full control over the HIDAPI build configuration.

Available CMake targets after {\ttfamily add\+\_\+subdirectory(hidapi)} {\itshape are the same as in case of standalone buildstandalone build}, and a few additional ones:\+
\begin{DoxyItemize}
\item {\ttfamily hidapi\+\_\+include} -\/ the interface library; {\ttfamily hidapi::\+include} is an alias of it;
\item {\ttfamily hidapi\+\_\+winapi} -\/ library target on Windows; {\ttfamily hidapi::\+winapi} is an alias of it;
\item {\ttfamily hidapi\+\_\+darwin} -\/ library target on mac\+OS; {\ttfamily hidapi::\+darwin} is an alias of it;
\item {\ttfamily hidapi\+\_\+libusb} -\/ library target for libusb backend; {\ttfamily hidapi::\+libusb} is an alias of it;
\item {\ttfamily hidapi\+\_\+hidraw} -\/ library target for hidraw backend; {\ttfamily hidapi::\+hidraw} is an alias of it;
\item {\ttfamily hidapi-\/libusb} -\/ an alias of {\ttfamily hidapi\+\_\+libusb} for compatibility with raw library name;
\item {\ttfamily hidapi-\/hidraw} -\/ an alias of {\ttfamily hidapi\+\_\+hidraw} for compatibility with raw library name;
\item {\ttfamily hidapi} -\/ an alias of {\ttfamily hidapi\+\_\+winapi} or {\ttfamily hidapi\+\_\+darwin} on Windows or mac\+OS respectfully.
\end{DoxyItemize}

If you\textquotesingle{}re a former (or present) user of Autotools build scripts for HIDAPI, or you\textquotesingle{}re a package manager maintainer and you\textquotesingle{}re often working with those -\/ you\textquotesingle{}re likely asking how to build HIDAPI with CMake and get both Shared and Static libraries (as would be done by Autotools:\+ {\ttfamily ./\+configure -\/-\/enable-\/static -\/-\/enable-\/shared ...}).

CMake doesn\textquotesingle{}t have such option of-\/the-\/box and it is decided not to introduce any manual CMake-\/level workarounds for HIDAPI on this matter.

If you want to mimic the Autotools behavior, it is possible by building/\+installing first the static version of the library and then shared version of the library. The installation folder ({\ttfamily CMAKE\+\_\+\+INSTALL\+\_\+\+PREFIX}) should point to the same directory for both variants, that way:\+
\begin{DoxyItemize}
\item both static and shared library binaries will be available and usable;
\item a single header file(s) for both of them;
\item Autotools/\+pkg-\/config ({\ttfamily .pc}) files will be generated and usable {\itshape as if} generated by Autotools natively and build configured with both {\ttfamily -\/enable-\/static -\/-\/enable-\/shared} options;
\item CMake package scripts will be generated and fully usable, but {\itshape only the last build installed}, i.\+e. if the last was installed Shared version of the binary -\/ CMake targets found by {\ttfamily find\+\_\+package(hidapi)} would point to a Shared binaries.
\end{DoxyItemize}

There is a historical discussion, why such solution is simplest/\+preferable:\+ \href{https://github.com/libusb/hidapi/issues/424}{\texttt{https:\+/\+/\+github.\+com/\+libusb/\+hidapi/\+issues/\+424}}\hypertarget{md__mono_runtime_2extern_2sdl_2src_2hidapi_2_b_u_i_l_d_8cmake_autotoc_md1644}{}\doxysubsubsubsection{\texorpdfstring{TL;DR/\+\+Sample}{TL;DR/\+\+Sample}}\label{md__mono_runtime_2extern_2sdl_2src_2hidapi_2_b_u_i_l_d_8cmake_autotoc_md1644}
