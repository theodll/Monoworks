\doxysubsection{Detailed Description}
\hypertarget{_s_d_l__storage_8h_autotoc_md1606}{}\doxysubsection{\texorpdfstring{Category\+Storage}{Category\+Storage}}\label{_s_d_l__storage_8h_autotoc_md1606}
The storage API is a high-\/level API designed to abstract away the portability issues that come up when using something lower-\/level (in SDL\textquotesingle{}s case, this sits on top of the \+[Filesystem]\+(Category\+Filesystem) and \+[IOStream]\+(Category\+IOStream) subsystems). It is significantly more restrictive than a typical filesystem API, for a number of reasons:\+


\begin{DoxyEnumerate}
\item {\bfseries{What to Access:\+}} A common pitfall with existing filesystem APIs is the assumption that all storage is monolithic. However, many other platforms (game consoles in particular) are more strict about what {\itshape type} of filesystem is being accessed; for example, game content and user data are usually two separate storage devices with entirely different characteristics (and possibly different low-\/level APIs altogether!).
\item {\bfseries{How to Access:\+}} Another common mistake is applications assuming that all storage is universally writeable -\/ again, many platforms treat game content and user data as two separate storage devices, and only user data is writeable while game content is read-\/only.
\item {\bfseries{When to Access:\+}} The most common portability issue with filesystem access is {\itshape timing} -\/ you cannot always assume that the storage device is always accessible all of the time, nor can you assume that there are no limits to how long you have access to a particular device.
\end{DoxyEnumerate}

Going over the bullet points again:\+


\begin{DoxyEnumerate}
\item {\bfseries{What to Access:\+}} This code accesses a global filesystem; game data and saves are all presumed to be in the current working directory (which may or may not be the game\textquotesingle{}s installation folder!).
\item {\bfseries{How to Access:\+}} This code assumes that content paths are writeable, and that save data is also writeable despite being in the same location as the game data.
\item {\bfseries{When to Access:\+}} This code assumes that they can be called at any time, since the filesystem is always accessible and has no limits on how long the filesystem is being accessed.
\end{DoxyEnumerate}

Due to these assumptions, the filesystem code is not portable and will fail under these common scenarios:\+


\begin{DoxyItemize}
\item The game is installed on a device that is read-\/only, both content loading and game saves will fail or crash outright
\item Game/\+\+User storage is not implicitly mounted, so no files will be found for either scenario when a platform requires explicitly mounting filesystems
\item Save data may not be safe since the I/\+O is not being flushed or validated, so an error occurring elsewhere in the program may result in missing/\+corrupted save data
\end{DoxyItemize}

Note the improvements that \doxylink{struct_s_d_l___storage}{SDL\+\_\+\+Storage} makes:\+


\begin{DoxyEnumerate}
\item {\bfseries{What to Access:\+}} This code explicitly reads from a title or user storage device based on the context of the function.
\item {\bfseries{How to Access:\+}} This code explicitly uses either a read or write function based on the context of the function.
\item {\bfseries{When to Access:\+}} This code explicitly opens the device when it needs to, and closes it when it is finished working with the filesystem.
\end{DoxyEnumerate}

The result is an application that is significantly more robust against the increasing demands of platforms and their filesystems!

A publicly available example of an \doxylink{struct_s_d_l___storage}{SDL\+\_\+\+Storage} backend is the \href{https://partner.steamgames.com/doc/features/cloud}{\texttt{Steam Cloud}} backend -\/ you can initialize Steamworks when starting the program, and then SDL will recognize that Steamworks is initialized and automatically use ISteam\+Remote\+Storage when the application opens user storage. More importantly, when you {\itshape open} storage it knows to begin a "{}batch"{} of filesystem operations, and when you {\itshape close} storage it knows to end and flush the batch. This is used by Steam to support \href{https://steamcommunity.com/groups/steamworks/announcements/detail/3142949576401813670}{\texttt{Dynamic Cloud Sync}} ; users can save data on one PC, put the device to sleep, and then continue playing on another PC (and vice versa) with the save data fully synchronized across all devices, allowing for a seamless experience without having to do full restarts of the program.\hypertarget{_s_d_l__storage_8h_autotoc_md1607}{}\doxysubsubsection{\texorpdfstring{Notes on valid paths}{Notes on valid paths}}\label{_s_d_l__storage_8h_autotoc_md1607}
All paths in the Storage API use Unix-\/style path separators (\textquotesingle{}/\+\textquotesingle{}). Using a different path separator will not work, even if the underlying platform would otherwise accept it. This is to keep code using the Storage API portable between platforms and Storage implementations and simplify app code.

Paths with relative directories ("{}."{} and "{}.."{}) are forbidden by the Storage API.

All valid UTF-\/8 strings (discounting the NULL terminator character and the \textquotesingle{}/\+\textquotesingle{} path separator) are usable for filenames, however, an underlying Storage implementation may not support particularly strange sequences and refuse to create files with those names, etc. Function interface for \doxylink{struct_s_d_l___storage}{SDL\+\_\+\+Storage}.

Apps that want to supply a custom implementation of \doxylink{struct_s_d_l___storage}{SDL\+\_\+\+Storage} will fill in all the functions in this struct, and then pass it to SDL\+\_\+\+Open\+Storage to create a custom \doxylink{struct_s_d_l___storage}{SDL\+\_\+\+Storage} object.

It is not usually necessary to do this; SDL provides standard implementations for many things you might expect to do with an \doxylink{struct_s_d_l___storage}{SDL\+\_\+\+Storage}.
