\hypertarget{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1155}{}\doxysubsection{\texorpdfstring{Purpose}{Purpose}}\label{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1155}
volk is a meta-\/loader for Vulkan. It allows you to dynamically load entrypoints required to use Vulkan without linking to vulkan-\/1.\+dll or statically linking Vulkan loader. Additionally, volk simplifies the use of Vulkan extensions by automatically loading all associated entrypoints. Finally, volk enables loading Vulkan entrypoints directly from the driver which can increase performance by skipping loader dispatch overhead.

volk is written in C89 and supports Windows, Linux, Android and mac\+OS (via Molten\+VK).\hypertarget{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1156}{}\doxysubsection{\texorpdfstring{Building}{Building}}\label{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1156}
There are multiple ways to use volk in your project:\+


\begin{DoxyEnumerate}
\item You can add {\ttfamily volk.\+c} to your build system. Note that the usual preprocessor defines that enable Vulkan\textquotesingle{}s platform-\/specific functions (VK\+\_\+\+USE\+\_\+\+PLATFORM\+\_\+\+WIN32\+\_\+\+KHR, VK\+\_\+\+USE\+\_\+\+PLATFORM\+\_\+\+XLIB\+\_\+\+KHR, VK\+\_\+\+USE\+\_\+\+PLATFORM\+\_\+\+MACOS\+\_\+\+MVK, etc) must be passed as desired to the compiler when building {\ttfamily volk.\+c}.
\item You can use provided CMake files, with the usage detailed below.
\item You can use volk in header-\/only fashion. Include {\ttfamily \doxylink{volk_8h_source}{volk.\+h}} wherever you want to use Vulkan functions. In exactly one source file, define {\ttfamily VOLK\+\_\+\+IMPLEMENTATION} before including {\ttfamily \doxylink{volk_8h_source}{volk.\+h}}. Do not build {\ttfamily volk.\+c} at all in this case -\/ however, {\ttfamily volk.\+c} must still be in the same directory as {\ttfamily \doxylink{volk_8h_source}{volk.\+h}}. This method of integrating volk makes it possible to set the platform defines mentioned above with arbitrary (preprocessor) logic in your code.
\end{DoxyEnumerate}\hypertarget{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1157}{}\doxysubsection{\texorpdfstring{Basic usage}{Basic usage}}\label{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1157}
To use volk, you have to include {\ttfamily \doxylink{volk_8h_source}{volk.\+h}} instead of {\ttfamily \doxylink{vulkan_8h_source}{vulkan/\+vulkan.\+h}}; this is necessary to use function definitions from volk.

If some files in your application include {\ttfamily \doxylink{vulkan_8h_source}{vulkan/\+vulkan.\+h}} and don\textquotesingle{}t include {\ttfamily \doxylink{volk_8h_source}{volk.\+h}}, this can result in symbol conflicts; consider defining {\ttfamily VK\+\_\+\+NO\+\_\+\+PROTOTYPES} when compiling code that uses Vulkan to make sure this doesn\textquotesingle{}t happen. It\textquotesingle{}s also important to make sure that {\ttfamily vulkan-\/1} is not linked into the application, as this results in symbol name conflicts as well.

To initialize volk, call this function first:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{VkResult\ volkInitialize();}

\end{DoxyCode}


This will attempt to load Vulkan loader from the system; if this function returns {\ttfamily VK\+\_\+\+SUCCESS} you can proceed to create Vulkan instance. If this function fails, this means Vulkan loader isn\textquotesingle{}t installed on your system.

After creating the Vulkan instance using Vulkan API, call this function:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{void}\ volkLoadInstance(VkInstance\ instance);}

\end{DoxyCode}


This function will load all required Vulkan entrypoints, including all extensions; you can use Vulkan from here on as usual.\hypertarget{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1158}{}\doxysubsection{\texorpdfstring{Optimizing device calls}{Optimizing device calls}}\label{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1158}
If you use volk as described in the previous section, all device-\/related function calls, such as {\ttfamily vk\+Cmd\+Draw}, will go through Vulkan loader dispatch code. This allows you to transparently support multiple Vk\+Device objects in the same application, but comes at a price of dispatch overhead which can be as high as 7\% depending on the driver and application.

To avoid this, you have two options:\+


\begin{DoxyEnumerate}
\item For applications that use just one Vk\+Device object, load device-\/related Vulkan entrypoints directly from the driver with this function:\+
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{void}\ volkLoadDevice(VkDevice\ device);}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item For applications that use multiple Vk\+Device objects, load device-\/related Vulkan entrypoints into a table:\+
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{void}\ volkLoadDeviceTable(\textcolor{keyword}{struct}\ \doxymbox{\hyperlink{struct_volk_device_table}{VolkDeviceTable}}*\ table,\ VkDevice\ device);}

\end{DoxyCode}


The second option requires you to change the application code to store one {\ttfamily \doxylink{struct_volk_device_table}{Volk\+Device\+Table}} per {\ttfamily Vk\+Device} and call functions from this table instead.

Device entrypoints are loaded using {\ttfamily vk\+Get\+Device\+Proc\+Addr}; when no layers are present, this commonly results in most function pointers pointing directly at the driver functions, minimizing the call overhead. When layers are loaded, the entrypoints will point at the implementations in the first applicable layer, so this is compatible with any layers including validation layers.

Since {\ttfamily volk\+Load\+Device} overwrites some function pointers with device-\/specific versions, you can choose to use {\ttfamily volk\+Load\+Instance\+Only} instead of {\ttfamily volk\+Load\+Instance}; when using table-\/based interface this can also help enforce the usage of the function tables as {\ttfamily volk\+Load\+Instance\+Only} will leave device-\/specific functions as {\ttfamily NULL}.\hypertarget{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1159}{}\doxysubsection{\texorpdfstring{CMake support}{CMake support}}\label{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1159}

The above example use {\ttfamily add\+\_\+subdirectory} to include volk into CMake\textquotesingle{}s build tree. This is a good choice if you copy the volk files into your project tree or as a git submodule.

volk also supports installation and config-\/file packages. Installation is disabled by default (so as to not pollute user projects with install rules), and can be enabled by passing {\ttfamily -\/DVOLK\+\_\+\+INSTALL=ON} to CMake. Once installed, do something like {\ttfamily find\+\_\+package(volk CONFIG REQUIRED)} in your project\textquotesingle{}s CMake\+Lists.\+txt. The imported volk targets are called {\ttfamily volk::\+volk} and {\ttfamily volk::\+volk\+\_\+headers}.\hypertarget{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1160}{}\doxysubsection{\texorpdfstring{Configuration}{Configuration}}\label{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1160}
By default, volk is compiled as a C library and exposes all Vulkan function pointers as globals. This can result in symbol conflicts if some libraries in the application are still linking to Vulkan libraries directly. While generally speaking it\textquotesingle{}s desirable to not mix \& match volk with direct usage of Vulkan -\/ for example, mixed usage means the application still links directly to Vulkan libraries and will fail to launch if Vulkan is not available on the user\textquotesingle{}s system -\/ it\textquotesingle{}s possible to enable {\ttfamily VOLK\+\_\+\+NAMESPACE} CMake option (or {\ttfamily VOLK\+\_\+\+NAMESPACE} define when building volk manually), which places all volk symbols into {\ttfamily volk:\+:\+} namespace. This requires compiling {\ttfamily volk.\+c} in C++ mode, which happens automatically when using CMake, but doesn\textquotesingle{}trequire any other changes.

Device level functions can be hidden by defining {\ttfamily VOLK\+\_\+\+NO\+\_\+\+DEVICE\+\_\+\+PROTOTYPES}. When using {\ttfamily volk\+Load\+Instance\+Only} and {\ttfamily volk\+Load\+Device\+Table} the device level functions are never loaded and when not used correctly would trigger a runtime error. By hiding the device prototypes mistakes can be checked by the compiler.\hypertarget{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1161}{}\doxysubsection{\texorpdfstring{License}{License}}\label{dir_960df8e10823cf9d133f9184ec32ecd8_autotoc_md1161}
