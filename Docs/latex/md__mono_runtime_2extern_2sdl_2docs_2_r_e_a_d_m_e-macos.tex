SDL is \href{README-platforms.md\#setuid}{\texttt{not designed to be used in setuid or setgid executables}}.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1356}{}\doxysection{\texorpdfstring{Using the Simple Direct\+Media Layer with a traditional Makefile}{Using the Simple Direct\+Media Layer with a traditional Makefile}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1356}
An existing build system for your SDL app has good chances to work almost unchanged on mac\+OS, as long as you link with the SDL framework. However, to produce a "{}real"{} Mac binary that you can distribute to users, you need to put the generated binary into a so called "{}bundle"{}, which is basically a fancy folder with a name like "{}\+My\+Cool\+Game.\+app"{}.

To get this build automatically, add something like the following rule to your Makefile.\+am:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{bundle\_contents\ =\ APP\_NAME.app/Contents}
\DoxyCodeLine{APP\_NAME\_bundle:\ EXE\_NAME}
\DoxyCodeLine{\ \ \ \ mkdir\ -\/p\ \$(bundle\_contents)/MacOS}
\DoxyCodeLine{\ \ \ \ mkdir\ -\/p\ \$(bundle\_contents)/Resources}
\DoxyCodeLine{\ \ \ \ echo\ "{}APPL????"{}\ >\ \$(bundle\_contents)/PkgInfo}
\DoxyCodeLine{\ \ \ \ \$(INSTALL\_PROGRAM)\ \$<\ \$(bundle\_contents)/MacOS/}

\end{DoxyCode}


You should replace {\ttfamily EXE\+\_\+\+NAME} with the name of the executable. {\ttfamily APP\+\_\+\+NAME} is what will be visible to the user in the Finder. Usually it will be the same as {\ttfamily EXE\+\_\+\+NAME} but capitalized. E.\+g. if {\ttfamily EXE\+\_\+\+NAME} is "{}testgame"{} then {\ttfamily APP\+\_\+\+NAME} usually is "{}\+Test\+Game"{}. You might also want to use {\ttfamily @PACKAGE@} to use the package name as specified in your configure.\+ac file.

If your project builds more than one application, you will have to do a bit more. For each of your target applications, you need a separate rule.

If you want the created bundles to be installed, you may want to add this rule to your Makefile.\+am:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{install-\/exec-\/hook:\ APP\_NAME\_bundle}
\DoxyCodeLine{\ \ \ \ rm\ -\/rf\ \$(DESTDIR)\$(prefix)/Applications/APP\_NAME.app}
\DoxyCodeLine{\ \ \ \ mkdir\ -\/p\ \$(DESTDIR)\$(prefix)/Applications/}
\DoxyCodeLine{\ \ \ \ cp\ -\/r\ \$<\ /\$(DESTDIR)\$(prefix)Applications/}

\end{DoxyCode}


This rule takes the Bundle created by the rule from step 3 and installs them into "{}\$(\+DESTDIR)\$(prefix)/\+\+Applications/\+"{}.

Again, if you want to install multiple applications, you will have to augment the make rule accordingly.

But beware! That is only part of the story! With the above, you end up with a barebones .app bundle, which is double-\/clickable from the Finder. But there are some more things you should do before shipping your product...


\begin{DoxyEnumerate}
\item You\textquotesingle{}ll need to copy the SDL framework into the Contents/\+\+Frameworks folder in your bundle, so it is included along with your application.
\item Add an \textquotesingle{}Info.\+plist\textquotesingle{} to your application. That is a special XML file which contains some meta-\/information about your application (like some copyright information, the version of your app, the name of an optional icon file, and other things). Part of that information is displayed by the Finder when you click on the .app, or if you look at the "{}\+Get Info"{} window. More information about Info.\+plist files can be found on Apple\textquotesingle{}s homepage.
\end{DoxyEnumerate}

The SDL Library is packaged as a framework bundle, an organized relocatable folder hierarchy of executable code, interface headers, and additional resources. For practical purposes, you can think of a framework as a more user and system-\/friendly shared library, whose library file behaves more or less like a standard UNIX shared library.

To build the framework, simply open the framework project and build it. By default, the framework bundle "{}\+SDL.\+framework"{} is installed in /\+\+Library/\+\+Frameworks. Therefore, the testers and project stationary expect it to be located there. However, it will function the same in any of the following locations:\+


\begin{DoxyItemize}

\item /\+\+Local/\+\+Library/\+\+Frameworks
\item /\+\+System/\+\+Library/\+\+Frameworks

Some things that may be of interest about how it all works...\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1367}{}\doxysubsection{\texorpdfstring{Working directory}{Working directory}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1367}
In SDL 1.\+2, the working directory of your SDL app is by default set to its parent, but this is no longer the case in SDL 2.\+0 and later. SDL2 does not change the working directory, which means it\textquotesingle{}ll be whatever the command line prompt that launched the program was using, or if launched by double-\/clicking in the Finder, it will be "{}/\+"{}, the {\itshape root of the filesystem}. Plan accordingly! You can use SDL\+\_\+\+Get\+Base\+Path() to find where the program is running from and chdir() there directly.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1368}{}\doxysubsection{\texorpdfstring{You have a Cocoa App!}{You have a Cocoa App!}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1368}
Your SDL app is essentially a Cocoa application. When your app starts up and the libraries finish loading, a Cocoa procedure is called, which sets up the working directory and calls your main() method. You are free to modify your Cocoa app with generally no consequence to SDL. You cannot, however, easily change the SDL window itself. Functionality may be added in the future to help this.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1369}{}\doxysubsection{\texorpdfstring{Raw Mouse Input}{Raw Mouse Input}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-macos_autotoc_md1369}
SDL uses the Game \doxylink{struct_controller}{Controller} framework\textquotesingle{}s GCMouse API to provide raw, unaccelerated mouse input in relative mode. This is ideal for games and applications requiring precise 1:\+1 mouse movement.

On versions older than mac\+OS 11.\+0 (Big Sur), SDL would fall back to NSEvent-\/based mouse input, which includes system mouse acceleration.

To use accelerated (system-\/scaled) mouse movement on mac\+OS 14.\+0+, set the hint:\+
