\chapter{Wayland }
\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland}{}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland}\index{Wayland@{Wayland}}
Wayland is a replacement for the X11 window system protocol and architecture and is favored over X11 by default in SDL3 for communicating with desktop compositors. It works well for the majority of applications, however, applications may encounter limitations or behavior that is different from other windowing systems.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland_autotoc_md1464}{}\doxysection{\texorpdfstring{Common issues:\+}{Common issues:\+}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland_autotoc_md1464}

\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland_autotoc_md1477}{}\doxysection{\texorpdfstring{Using custom Wayland windowing protocols with SDL windows}{Using custom Wayland windowing protocols with SDL windows}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland_autotoc_md1477}
Under normal operation, an {\ttfamily \doxylink{struct_s_d_l___window}{SDL\+\_\+\+Window}} corresponds to an XDG toplevel window, which provides a standard desktop window. If an application wishes to use a different windowing protocol with an SDL window (e.\+g. wlr\+\_\+layer\+\_\+shell) while still having SDL handle input and rendering, it needs to create a custom, roleless surface and attach that surface to its own toplevel window.

This is done by using {\ttfamily SDL\+\_\+\+Create\+Window\+With\+Properties()} and setting the {\ttfamily SDL\+\_\+\+PROP\+\_\+\+WINDOW\+\_\+\+CREATE\+\_\+\+WAYLAND\+\_\+\+SURFACE\+\_\+\+ROLE\+\_\+\+CUSTOM\+\_\+\+BOOLEAN} property to {\ttfamily true}. Once the window has been successfully created, the {\ttfamily wl\+\_\+display} and {\ttfamily wl\+\_\+surface} objects can then be retrieved from the {\ttfamily SDL\+\_\+\+PROP\+\_\+\+WINDOW\+\_\+\+WAYLAND\+\_\+\+DISPLAY\+\_\+\+POINTER} and {\ttfamily SDL\+\_\+\+PROP\+\_\+\+WINDOW\+\_\+\+WAYLAND\+\_\+\+SURFACE\+\_\+\+POINTER} properties respectively.

Surfaces don\textquotesingle{}t receive any size change notifications, so if an application changes the window size, it must inform SDL that the surface size has changed by calling SDL\+\_\+\+Set\+Window\+Size() with the new dimensions.

Custom surfaces will automatically handle scaling internally if the window was created with the {\ttfamily SDL\+\_\+\+PROP\+\_\+\+WINDOW\+\_\+\+CREATE\+\_\+\+HIGH\+\_\+\+PIXEL\+\_\+\+DENSITY\+\_\+\+BOOLEAN} property set to {\ttfamily true}. In this case, applications should not manually attach viewports or change the surface scale value, as SDL will handle this internally. Calls to {\ttfamily SDL\+\_\+\+Set\+Window\+Size()} should use the logical size of the window, and {\ttfamily SDL\+\_\+\+Get\+Window\+Size\+In\+Pixels()} should be used to query the size of the backbuffer surface in pixels. If this property is not set or is {\ttfamily false}, applications can attach their own viewports or change the surface scale manually, and the SDL backend will not interfere or change any values internally. In this case, calls to {\ttfamily SDL\+\_\+\+Set\+Window\+Size()} should pass the requested surface size in pixels, not the logical window size, as no scaling calculations will be done internally.

All window functions that control window state aside from {\ttfamily SDL\+\_\+\+Set\+Window\+Size()} are no-\/ops with custom surfaces.

Please see the minimal example in {\ttfamily tests/\+testwaylandcustom.\+c} for an example of how to use a custom, roleless surface and attach it to an application-\/managed toplevel window.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland_autotoc_md1478}{}\doxysection{\texorpdfstring{Importing external surfaces into SDL windows}{Importing external surfaces into SDL windows}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-wayland_autotoc_md1478}
Wayland windows and surfaces are more intrinsically tied to the client library than other windowing systems, therefore, when importing surfaces, it is necessary for both SDL and the application or toolkit to use the same {\ttfamily wl\+\_\+display} object. This can be set/\+queried via the global {\ttfamily SDL\+\_\+\+PROP\+\_\+\+GLOBAL\+\_\+\+VIDEO\+\_\+\+WAYLAND\+\_\+\+WL\+\_\+\+DISPLAY\+\_\+\+POINTER} property. To import an external {\ttfamily wl\+\_\+display}, set this property before initializing the SDL video subsystem, and read the value to export the internal {\ttfamily wl\+\_\+display} after the video subsystem has been initialized. Setting this property after the video subsystem has been initialized has no effect, and reading it when the video subsystem is uninitialized will either return the user provided value, if one was set while in the uninitialized state, or NULL.

Once this is done, and the application has created or obtained the {\ttfamily wl\+\_\+surface} to be wrapped in an {\ttfamily \doxylink{struct_s_d_l___window}{SDL\+\_\+\+Window}}, the window is created with {\ttfamily SDL\+\_\+\+Create\+Window\+With\+Properties()} with the {\ttfamily SDL\+\_\+\+PROP\+\_\+\+WINDOW\+\_\+\+CREATE\+\_\+\+WAYLAND\+\_\+\+WL\+\_\+\+SURFACE\+\_\+\+POINTER} property to set to the {\ttfamily wl\+\_\+surface} object that is to be imported by SDL.

SDL receives no notification regarding size changes on external surfaces or toplevel windows, so if the external surface needs to be resized, SDL must be informed by calling SDL\+\_\+\+Set\+Window\+Size() with the new dimensions.

If desired, SDL can automatically handle the scaling for the surface by setting the {\ttfamily SDL\+\_\+\+PROP\+\_\+\+WINDOW\+\_\+\+CREATE\+\_\+\+HIGH\+\_\+\+PIXEL\+\_\+\+DENSITY\+\_\+\+BOOLEAN} property to {\ttfamily true}, however, if the surface being imported already has, or will have, a viewport/\+fractional scale manager attached to it by the application or an external toolkit, a protocol violation will result. Avoid setting this property if importing surfaces from toolkits such as Qt or GTK.

If the window is flagged as high pixel density, calls to {\ttfamily SDL\+\_\+\+Set\+Window\+Size()} should pass the logical size of the window and {\ttfamily SDL\+\_\+\+Get\+Window\+Size\+In\+Pixels()} should be used to retrieve the backbuffer size in pixels. Otherwise, calls to {\ttfamily SDL\+\_\+\+Set\+Window\+Size()} should pass the requested surface size in pixels, not the logical window size, as no scaling calculations will be done internally.

All window functions that control window state aside from {\ttfamily SDL\+\_\+\+Set\+Window\+Size()} are no-\/ops with external surfaces.
