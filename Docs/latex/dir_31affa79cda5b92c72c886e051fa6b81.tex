\doxysubsection{Detailed Description}
Net\+BSD maps every {\ttfamily uhidev} device to one or more {\ttfamily uhid} devices. Each {\ttfamily uhid} device only supports one report ID. The parent device {\ttfamily uhidev} creates one {\ttfamily uhid} device per report ID found in the hardware\textquotesingle{}s report descriptor.

In the event there are no report ID(s) found within the report descriptor, only one {\ttfamily uhid} device with a report ID of {\ttfamily 0} is created.

In order to remain compatible with existing {\ttfamily hidapi} APIs, all the {\ttfamily uhid} devices created by the parent {\ttfamily uhidev} device must be opened under the same {\ttfamily \doxylink{hidapi_8h_aa6da74d5686d198dd3e5440e60088fcc}{hid\+\_\+device}} instance to ensure that we can route reports to their appropriate {\ttfamily uhid} device.

Internally the {\ttfamily uhid} driver will insert the report ID as needed so we must also omit the report ID in any situation where the {\ttfamily hidapi} API expects it to be included in the report data stream.

Given the design of {\ttfamily uhid}, it must be augmented with extra platform specific APIs to ensure that the exact relationship between {\ttfamily uhidev} devices and {\ttfamily uhid} devices can be determined.

The Net\+BSD implementation does this via the {\ttfamily drvctl} kernel driver. At present there is no known way to do this on Open\+BSD for a {\ttfamily uhid} implementation to be at the same level as the Net\+BSD one.
