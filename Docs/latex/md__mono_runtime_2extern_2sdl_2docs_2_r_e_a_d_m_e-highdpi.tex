SDL 3.\+0 has new support for high DPI displays. Interfaces provided by SDL uses the platform\textquotesingle{}s native coordinates unless otherwise specified.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-highdpi_autotoc_md1332}{}\doxysubsection{\texorpdfstring{Explanation}{Explanation}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-highdpi_autotoc_md1332}
Displays now have a content display scale, which is the expected scale for content based on the DPI settings of the display. For example, a 4K display might have a 2.\+0 (200\%) display scale, which means that the user expects UI elements to be twice as big on this display, to aid in readability. You can query the display content scale using {\ttfamily SDL\+\_\+\+Get\+Display\+Content\+Scale()}, and when this changes you get an {\ttfamily SDL\+\_\+\+EVENT\+\_\+\+WINDOW\+\_\+\+DISPLAY\+\_\+\+SCALE\+\_\+\+CHANGED} event.

The window size is now distinct from the window pixel size, and the ratio between the two is the window pixel density. If the window is created with the {\ttfamily SDL\+\_\+\+WINDOW\+\_\+\+HIGH\+\_\+\+PIXEL\+\_\+\+DENSITY} flag, SDL will try to match the native pixel density for the display, otherwise it will try to have the pixel size match the window size. You can query the window pixel density using {\ttfamily SDL\+\_\+\+Get\+Window\+Pixel\+Density()}. You can query the window pixel size using {\ttfamily SDL\+\_\+\+Get\+Window\+Size\+In\+Pixels()}, and when this changes you get an {\ttfamily SDL\+\_\+\+EVENT\+\_\+\+WINDOW\+\_\+\+PIXEL\+\_\+\+SIZE\+\_\+\+CHANGED} event. You are guaranteed to get a {\ttfamily SDL\+\_\+\+EVENT\+\_\+\+WINDOW\+\_\+\+PIXEL\+\_\+\+SIZE\+\_\+\+CHANGED} event when a window is created and resized, and you can use this event to create and resize your graphics context for the window.

The window has a display scale, which is the scale from the pixel resolution to the desired content size, e.\+g. the combination of the pixel density and the content scale. For example, a 3840x2160 window displayed at 200\% on Windows, and a 1920x1080 window with the high density flag on a 2x display on mac\+OS will both have a pixel size of 3840x2160 and a display scale of 2.\+0. You can query the window display scale using {\ttfamily SDL\+\_\+\+Get\+Window\+Display\+Scale()}, and when this changes you get an {\ttfamily SDL\+\_\+\+EVENT\+\_\+\+WINDOW\+\_\+\+DISPLAY\+\_\+\+SCALE\+\_\+\+CHANGED} event.\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-highdpi_autotoc_md1333}{}\doxysubsection{\texorpdfstring{Numeric example}{Numeric example}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-highdpi_autotoc_md1333}

Observe the difference between the approaches taken by mac\+OS and Windows:\+
\begin{DoxyItemize}
\item The Windows and Android coordinate system always deals in physical device pixels, high DPI support is achieved by providing a content scale that tells the developer to draw objects larger. Ignoring this scale factor results in graphics appearing tiny.
\item The mac\+OS and i\+OS coordinate system always deals in window coordinates, high DPI support is achieved by providing an optional flag for the developer to request more pixels. Omitting this flag results in graphics having low detail.
\item On Linux, X11 uses a similar approach to Windows and Wayland uses a similar approach to mac\+OS.
\end{DoxyItemize}\hypertarget{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-highdpi_autotoc_md1334}{}\doxysubsection{\texorpdfstring{Solution}{Solution}}\label{md__mono_runtime_2extern_2sdl_2docs_2_r_e_a_d_m_e-highdpi_autotoc_md1334}
Proper high DPI support takes into account both the content scale and the pixel density.

First, you\textquotesingle{}d create your window with the {\ttfamily SDL\+\_\+\+WINDOW\+\_\+\+HIGH\+\_\+\+PIXEL\+\_\+\+DENSITY} flag, assuming you want the highest detail possible. Then you\textquotesingle{}d get the window display scale to see how much your UI elements should be enlarged to be readable.

If you\textquotesingle{}re using the SDL 2D renderer, SDL provides the function {\ttfamily SDL\+\_\+\+Convert\+Event\+To\+Render\+Coordinates()} to convert mouse coordinates between window coordinates and rendering coordinates, and the more general functions {\ttfamily SDL\+\_\+\+Render\+Coordinates\+From\+Window()} and {\ttfamily SDL\+\_\+\+Render\+Coordinates\+To\+Window()} to do other conversion between them.

If you\textquotesingle{}re not using the 2D renderer, you can implement this yourself using {\ttfamily SDL\+\_\+\+Get\+Window\+Pixel\+Density()} as scale factor to convert from window coordinates to pixels.

Finally you\textquotesingle{}ll want to test on both Windows and mac\+OS if possible to make sure your high DPI support works in all environments.
