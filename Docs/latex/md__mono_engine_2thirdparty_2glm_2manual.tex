\end{DoxyCode}

Every object type has the property called alignment requirement, which is an integer value (of type {\ttfamily std::\+size\+\_\+t}, always a power of 2) representing the number of bytes between successive addresses at which objects of this type can be allocated. The alignment requirement of a type can be queried with alignof or {\ttfamily std::\+alignment\+\_\+of}. The pointer alignment function {\ttfamily std::\+align} can be used to obtain a suitably-\/aligned pointer within some buffer, and {\ttfamily std::\+aligned\+\_\+storage} can be used to obtain suitably-\/aligned storage.

Each object type imposes its alignment requirement on every object of that type; stricter alignment (with larger alignment requirement) can be requested using C++11 {\ttfamily alignas}.

In order to satisfy alignment requirements of all non-\/static members of a class, padding may be inserted after some of its members.

GLM supports both packed and aligned types. Packed types allow filling data structure without inserting extra padding. Aligned GLM types align addresses based on the size of the value type of a GLM type.

The GLSL function \textquotesingle{}not\textquotesingle{} is a keyword in C++. To prevent name collisions and ensure a consistent API, the name {\ttfamily not\textbackslash{}\+\_\+} (note the underscore) is used instead.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md168}{}\doxysubsubsection{\texorpdfstring{\label{_section6_2}%
 6.\+2. Precision qualifiers support}{\label{_section6_2}%
 6.\+2. Precision qualifiers support}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md168}
GLM supports GLSL precision qualifiers through prefixes instead of qualifiers. For example, GLM exposes \textbackslash{}verb\texorpdfstring{$\vert$}{|}lowp\+\_\+vec4\texorpdfstring{$\vert$}{|}, \textbackslash{}verb\texorpdfstring{$\vert$}{|}mediump\+\_\+vec4\texorpdfstring{$\vert$}{|} and \textbackslash{}verb\texorpdfstring{$\vert$}{|}highp\+\_\+vec4\texorpdfstring{$\vert$}{|} as variations of \textbackslash{}verb\texorpdfstring{$\vert$}{|}vec4\texorpdfstring{$\vert$}{|}.

Similarly to GLSL, GLM precision qualifiers are used to trade precision of operations in term of \href{http://en.wikipedia.org/wiki/Unit_in_the_last_place}{\texttt{ULPs}} for better performance. By default, all the types use high precision.

\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md171}{}\doxysubsubsection{\texorpdfstring{\label{_section7_1}%
 7.\+1 Why GLM follows GLSL specification and conventions?}{\label{_section7_1}%
 7.\+1 Why GLM follows GLSL specification and conventions?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md171}
Following GLSL conventions is a really strict policy of GLM. It has been designed following the idea that everyone does its own math library with his own conventions. The idea is that brilliant developers (the Open\+GL ARB) worked together and agreed to make GLSL. Following GLSL conventions is a way to find consensus. Moreover, basically when a developer knows GLSL, he knows GLM.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md172}{}\doxysubsubsection{\texorpdfstring{\label{_section7_2}%
 7.\+2. Does GLM run GLSL program?}{\label{_section7_2}%
 7.\+2. Does GLM run GLSL program?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md172}
No, GLM is a C++ implementation of a subset of GLSL.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md173}{}\doxysubsubsection{\texorpdfstring{\label{_section7_3}%
 7.\+3. Does a GLSL compiler build GLM codes?}{\label{_section7_3}%
 7.\+3. Does a GLSL compiler build GLM codes?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md173}
No, this is not what GLM intends to do.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md174}{}\doxysubsubsection{\texorpdfstring{\label{_section7_4}%
 7.\+4. Should I use \textquotesingle{}GTX\textquotesingle{} extensions?}{\label{_section7_4}%
 7.\+4. Should I use \textquotesingle{}GTX\textquotesingle{} extensions?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md174}
GTX extensions are qualified to be experimental extensions. In GLM this means that these extensions might change from version to version without any restriction. In practice, it doesn\textquotesingle{}t really change except time to time. GTC extensions are stabled, tested and perfectly reliable in time. Many GTX extensions extend GTC extensions and provide a way to explore features and implementations and APIs and then are promoted to GTC extensions. This is fairly the way Open\+GL features are developed; through extensions.

Starting with GLM 0.\+9.\+9, to use experimental extensions, an application must define GLM\+\_\+\+ENABLE\+\_\+\+EXPERIMENTAL.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md175}{}\doxysubsubsection{\texorpdfstring{\label{_section7_5}%
 7.\+5. Where can I ask my questions?}{\label{_section7_5}%
 7.\+5. Where can I ask my questions?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md175}
A good place is \href{http://stackoverflow.com/search?q=GLM}{\texttt{stackoverflow}} using the GLM tag.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md176}{}\doxysubsubsection{\texorpdfstring{\label{_section7_6}%
 7.\+6. Where can I find the documentation of extensions?}{\label{_section7_6}%
 7.\+6. Where can I find the documentation of extensions?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md176}
The Doxygen generated documentation includes a complete list of all extensions available. Explore this \href{http://glm.g-truc.net/0.9.9/api/modules.html}{\texttt{{\itshape API documentation}}} to get a complete view of all GLM capabilities!\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md177}{}\doxysubsubsection{\texorpdfstring{\label{_section7_7}%
 7.\+7. Should I use \textquotesingle{}using namespace glm;\textquotesingle{}?}{\label{_section7_7}%
 7.\+7. Should I use \textquotesingle{}using namespace glm;\textquotesingle{}?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md177}
NO! Chances are that if using namespace glm; is called, especially in a header file, name collisions will happen as GLM is based on GLSL which uses common tokens for types and functions. Avoiding using namespace glm; will a higher compatibility with third party library and SDKs.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md178}{}\doxysubsubsection{\texorpdfstring{\label{_section7_8}%
 7.\+8. Is GLM fast?}{\label{_section7_8}%
 7.\+8. Is GLM fast?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md178}
GLM is mainly designed to be convenient and that\textquotesingle{}s why it is written against the GLSL specification.

Following the Pareto principle where 20\% of the code consumes 80\% of the execution time, GLM operates perfectly on the 80\% of the code that consumes 20\% of the performances. Furthermore, thanks to the lowp, mediump and highp qualifiers, GLM provides approximations which trade precision for performance. Finally, GLM can automatically produce SIMD optimized code for functions of its implementation.

However, on performance critical code paths, we should expect that dedicated algorithms should be written to reach peak performance.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md179}{}\doxysubsubsection{\texorpdfstring{\label{_section7_9}%
 7.\+9. When I build with Visual C++ with /\+\+W4 warning level, I have warnings...}{\label{_section7_9}%
 7.\+9. When I build with Visual C++ with /\+\+W4 warning level, I have warnings...}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md179}
You should not have any warnings even in {\ttfamily /\+\+W4} mode. However, if you expect such level for your code, then you should ask for the same level to the compiler by at least disabling the Visual C++ language extensions ({\ttfamily /\+\+Za}) which generates warnings when used. If these extensions are enabled, then GLM will take advantage of them and the compiler will generate warnings.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md180}{}\doxysubsubsection{\texorpdfstring{\label{_section7_10}%
 7.\+10. Why some GLM functions can crash because of division by zero?}{\label{_section7_10}%
 7.\+10. Why some GLM functions can crash because of division by zero?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md180}
GLM functions crashing is the result of a domain error. Such behavior follows the precedent set by C and C++\textquotesingle{}s standard library. For example, it\textquotesingle{}s a domain error to pass a null vector (all zeroes) to \doxylink{group__core__func__geometric_ga5c6c1379961d4589035a390856f2d66c}{glm::\+normalize} function, or to pass a negative number into std::\+sqrt.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md181}{}\doxysubsubsection{\texorpdfstring{\label{_section7_11}%
 7.\+11. What unit for angles is used in GLM?}{\label{_section7_11}%
 7.\+11. What unit for angles is used in GLM?}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md181}
GLSL is using radians but GLU is using degrees to express angles. This has caused GLM to use inconsistent units for angles. Starting with GLM 0.\+9.\+6, all GLM functions are using radians. For more information, follow the \href{http://www.g-truc.net/post-0693.html\#menu}{\texttt{link}}.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md182}{}\doxysubsubsection{\texorpdfstring{\label{_section7_12}%
 7.\+12. Windows headers cause build errors...}{\label{_section7_12}%
 7.\+12. Windows headers cause build errors...}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md182}
Some Windows headers define min and max as macros which may cause compatibility with third party libraries such as GLM. It is highly recommended to \href{http://stackoverflow.com/questions/4913922/possible-problems-with-nominmax-on-visual-c}{\texttt{{\ttfamily define NOMINMAX}}} before including Windows headers to workaround this issue. To workaround the incompatibility with these macros, GLM will systematically undef these macros if they are defined.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md183}{}\doxysubsubsection{\texorpdfstring{\label{_section7_13}%
 7.\+13. Constant expressions support}{\label{_section7_13}%
 7.\+13. Constant expressions support}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md183}
GLM has some C++ \href{http://en.cppreference.com/w/cpp/language/constexpr}{\texttt{constant expressions}} support. However, GLM automatically detects the use of SIMD instruction sets through compiler arguments to populate its implementation with SIMD intrinsics. Unfortunately, GCC and Clang don\textquotesingle{}t support SIMD intrinsics as constant expressions. To allow constant expressions on all vectors and matrices types, define {\ttfamily GLM\+\_\+\+FORCE\+\_\+\+PURE} before including GLM headers.

This tutorial will show us how to successfully contribute a bug-\/fix to GLM using Git\+Hub\textquotesingle{}s Pull Request workflow.

We will be typing git commands in the Terminal. Mac and Linux users may have git pre-\/installed. You can download git from \href{http://git-scm.com/downloads}{\texttt{here}}.

The tutorial assumes you have some basic understanding of git concepts -\/ repositories, branches, commits, etc. Explaining it all from scratch is beyond the scope of this tutorial. Some good links to learn git basics are:\+ \href{http://git-scm.com/book/en/Getting-Started-Git-Basics}{\texttt{Link 1}}, \href{https://www.atlassian.com/git/tutorial/git-basics}{\texttt{Link 2}}.\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md194}{}\doxysubsubsubsection{\texorpdfstring{Step 1:\+ Setup our GLM Fork}{Step 1:\+ Setup our GLM Fork}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md194}
We will make our changes in our own copy of the GLM repository. On the GLM Git\+Hub repo and we press the Fork button. We need to download a copy of our fork to our local machine. In the terminal, type:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ clone\ <our-\/repository-\/fork-\/git-\/url>}

\end{DoxyCode}


This will clone our fork repository into the current folder.

We can find our repository git url on the Github repository page. The url looks like this:\+ {\ttfamily \href{https://github.com/}{\texttt{https:\+/\+/\+github.\+com/\+}}\texorpdfstring{$<$}{<}our-\/username\texorpdfstring{$>$}{>}/\+\texorpdfstring{$<$}{<}repository-\/name\texorpdfstring{$>$}{>}.git}\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md195}{}\doxysubsubsubsection{\texorpdfstring{Step 2:\+ Synchronizing our fork}{Step 2:\+ Synchronizing our fork}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md195}
We can use the following command to add {\ttfamily upstream} (original project repository) as a remote repository so that we can fetch the latest GLM commits into our branch and keep our forked copy is synchronized.


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ remote\ add\ upstream\ https://github.com/g-\/truc/glm.git}

\end{DoxyCode}


To synchronize our fork to the latest commit in the GLM repository, we can use the following command:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ fetch\ upstream}

\end{DoxyCode}

\begin{DoxyCode}{0}
\DoxyCodeLine{git\ merge\ upstream/master}

\end{DoxyCode}

Our fork is now setup and we are ready to modify GLM to fix a bug.

It\textquotesingle{}s a good practice to make changes in our fork in a separate branch than the master branch because we can submit only one pull request per branch.

Before creating a new branch, it\textquotesingle{}s best to synchronize our fork and then create a new branch from the latest master branch.

If we are not on the master branch, we should switch to it using:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ checkout\ master}

\end{DoxyCode}


To create a new branch called {\ttfamily bugfix}, we use:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ branch\ bugfix}

\end{DoxyCode}


Once the code changes for the fix is done, we need to commit the changes:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ commit\ -\/m\ "{}Resolve\ the\ issue\ that\ caused\ problem\ with\ a\ specific\ fix\ \#432"{}}

\end{DoxyCode}


The commit message should be as specific as possible and finished by the bug number in the \href{https://github.com/g-truc/glm/issues}{\texttt{GLM Git\+Hub issue page}}.

Finally, we need to push our changes in our branch to our Git\+Hub fork using:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ push\ origin\ bugfix}

\end{DoxyCode}


Some things to keep in mind for a pull request:\+


\begin{DoxyItemize}
\item Keep it minimal:\+ Try to make the minimum required changes to fix the issue. If we have added any debugging code, we should remove it.
\item A fix at a time:\+ The pull request should deal with one issue at a time only, unless two issues are so interlinked they must be fixed together.
\item Write a test:\+ GLM is largely unit tests. \doxylink{struct_unit}{Unit} tests are in {\ttfamily glm/\+test} directory. We should also add tests for the fixes we provide to ensure future regression doesn\textquotesingle{}t happen.
\item No whitespace changes:\+ Avoid unnecessary formatting or whitespace changes in other parts of the code. Be careful with auto-\/format options in the code editor which can cause wide scale formatting changes.
\item Follow GLM Code StyleGLM Code Style
\end{DoxyItemize}for consistency.
\begin{DoxyItemize}
\item Tests passes:\+ Make sure GLM build and tests don\textquotesingle{}t fail because of the changes.
\end{DoxyItemize}\hypertarget{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md197}{}\doxysubsubsubsection{\texorpdfstring{Step 4:\+ Submitting a Pull Request}{Step 4:\+ Submitting a Pull Request}}\label{md__mono_engine_2thirdparty_2glm_2manual_autotoc_md197}
We need to submit a pull request from the {\ttfamily bugfix} branch to GLM\textquotesingle{}s master branch.

On the fork github page, we can click on the {\itshape Pull Request} button. Then we can describe our pull request. Finally we press {\itshape Send Pull Request}.

Please be patient and give them some time to go through it.

The pull request review may suggest additional changes. So we can make those changes in our branch, and push those changes to our fork repository. Our pull request will always include the latest changes in our branch on Git\+Hub, so we don\textquotesingle{}t need to resubmit the pull request.

Once your changes have been accepted, a project maintainer will merge our pull request.
