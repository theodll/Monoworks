\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_f_a_q_autotoc_md543}{}\doxysubsubsection{\texorpdfstring{Q. What is this library called?}{Q. What is this library called?}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_f_a_q_autotoc_md543}
{\bfseries{This library is called Dear Im\+Gui}}. Please refer to it as Dear Im\+Gui (not Im\+Gui, not IMGUI).

(The library misleadingly started its life in 2014 as "{}\+Im\+Gui"{} due to the fact that I didn\textquotesingle{}t give it a proper name when I released 1.\+0, and had no particular expectation that it would take off. However, the term IMGUI (immediate-\/mode graphical user interface) was coined before and is being used in variety of other situations e.\+g. Unity uses it own implementation of the IMGUI paradigm. To reduce the ambiguity without affecting existing code bases, I have decided in December 2015 a fully qualified name "{}\+Dear Im\+Gui"{} for this library.\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_f_a_q_autotoc_md544}{}\doxysubsubsubsubsection{\texorpdfstring{\protect\doxysectlink{index}{Return to Index}{0}Return to Index}{{Return to Index}Return to Index}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_f_a_q_autotoc_md544}

Here\textquotesingle{}s a very simplified comparison between the approach taken by Dear Im\+Gui vs traditional toolkits:\+

UI layout is fully dynamic and can change at any time.~\newline
UI is generally emitted programmatically, which empowers reflecting a dynamic set of data.  &UI layout is mostly static.~\newline

UI library stores minimal amounts of data. At one point in time, it typically doesn\textquotesingle{}t know or remember which other widgets are displayed and which widgets are coming next. As a result, certain layout features (alignment, resizing) are not as easy to implement or require ad-\/hoc code.  &UI library stores entire widgets tree and state. Data is naturally always synchronized.  &Use callback/\+signal/\+slot for synchronizing data (error-\/prone).  \\
API is simple and easy to learn. In particular, doing basic things is very easy.  &API is more complex and specialized.  \\

 This is only meant to give you a intuitive feeling of the main differences, but pros \& cons go deeper than that.

Some of those properties are typically associated to the umbrella term "{}\+IMGUI"{}, but the term has no simple and well-\/agreed definition. There are many erroneous statements and misunderstandings with what IMGUI means. It is partly caused by the fact that most popular IMGUI implementations (including Dear Im\+Gui) have originated from game industry needs and have targeted specific use cases, causing people to conflate IMGUI properties with what a specific library does. However, it is perfectly possible to implement an IMGUI library that would have very different properties than e.\+g. Dear Im\+Gui. My take on defining what an IMGUI is:\+

{\bfseries{IMGUI refers to the API:\+ literally the interface between the application and the UI system.}}
\begin{DoxyItemize}
\item An IMGUI API favors the application code owning its data and being the single source of truth for it.
\item An IMGUI API tries to minimize the application having to retain/\+manage data related to the UI system.
\item An IMGUI API tries to minimize the UI system having to retain/\+manage data related to the application.
\item Synchronization between application data and UI data is natural and less error-\/prone.
\end{DoxyItemize}

{\bfseries{IMGUI does NOT refer to the implementation. Whatever happens inside the UI library code doesn\textquotesingle{}t matter.}} ~\newline

You can read the {\ttfamily io.\+Want\+Capture\+Mouse}, {\ttfamily io.\+Want\+Capture\+Keyboard} and {\ttfamily io.\+Want\+Text\+Input} flags from the \doxylink{struct_im_gui_i_o}{Im\+Gui\+IO} structure.
\begin{DoxyItemize}
\item When {\ttfamily io.\+Want\+Capture\+Mouse} is set, you need to discard/\+hide the mouse inputs from your underlying application.
\item When {\ttfamily io.\+Want\+Capture\+Keyboard} is set, you need to discard/\+hide the keyboard inputs from your underlying application.
\item When {\ttfamily io.\+Want\+Text\+Input} is set, you can notify your OS/\+engine to popup an on-\/screen keyboard, if available (e.\+g. on a mobile phone, or console OS).
\end{DoxyItemize}

Important:\+ you should always pass your mouse/\+keyboard inputs to Dear Im\+Gui, regardless of the value {\ttfamily io.\+Want\+Capture\+Mouse}/\+{\ttfamily io.\+Want\+Capture\+Keyboard}. This is because e.\+g. we need to detect that you clicked in the void to unfocus its own windows, and other reasons.

\begin{DoxyItemize}
\item Widgets labels are also used to compute Widgets unique identifiers.
\item Unique identifiers are hashes of the label + of the parent scope (e.\+g. parent window or parent tree node labels).

\item You can use {\ttfamily "{}\#\#something"{}} in a label to append to the identifier without making it visible.
\item You can use {\ttfamily "{}\#\#\#something"{}} in a label to make the identifier ignore the visible part.

Dear Im\+Gui internally needs to uniquely identify UI elements. Elements that are typically not clickable (such as calls to Text() functions) don\textquotesingle{}t need an ID. Interactive widgets (such as calls to Button() functions) need a unique ID.

{\bfseries{Unique IDs are used internally to track active widgets and occasionally associate state to widgets.~\newline
 Unique IDs are implicitly built from the hash of multiple elements that identify the "{}path"{} to the UI element.}}

{\bfseries{Short explanation:\+}}
\begin{DoxyItemize}
\item You may use functions such as {\ttfamily Im\+Gui::\+\+Image()}, {\ttfamily Im\+Gui::\+\+Image\+Button()} or lower-\/level {\ttfamily Im\+Draw\+List::\+\+Add\+Image()} to emit draw calls that will use your own textures.
\item To load and display your own textures, refer to \href{https://github.com/ocornut/imgui/wiki/Image-Loading-and-Displaying-Examples}{\texttt{Image Loading and Displaying Examples}} on the \href{https://github.com/ocornut/imgui/wiki}{\texttt{Wiki}}.
\item Actual textures are identified in a way that is up to the user/\+engine. Those identifiers are stored and passed as an opaque Im\+Texture\+ID value.
\item By default Im\+Texture\+ID can store up to 64-\/bits. You may {\ttfamily \#define} it to a custom type/\+structure if you need.
\item Loading image files from the disk and turning them into a texture is not within the scope of Dear Im\+Gui (for a good reason), but the examples linked above may be useful references.
\end{DoxyItemize}

{\bfseries{Details:\+}}

1.\+92 introduced {\ttfamily \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref}} in June 2025.
\begin{DoxyItemize}
\item All drawing functions using {\ttfamily Im\+Texture\+ID} were changed to use {\ttfamily \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref}}.
\item You can trivially create a {\ttfamily \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref}} from a {\ttfamily Im\+Texture\+ID}.
\item {\bfseries{If you use Image functions with textures that you have loaded/\+created yourself, you will mostly likely only ever store/\+manipulate {\ttfamily Im\+Texture\+ID} and then pass them as {\ttfamily \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref}}.}}
\item You only NEED to manipulate {\ttfamily \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref}} when dealing with textures managed by the backend itself, aka mainly the atlas texture for now.
\item We intentionally do not provide an implicit {\ttfamily \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref}} -\/\texorpdfstring{$>$}{>} {\ttfamily Im\+Texture\+ID} cast operator because it is technically lossy to convert \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref} to Im\+Texture\+ID before rendering.
\end{DoxyItemize}

{\bfseries{Please read documentations or tutorials on your graphics API to understand how to display textures on the screen before moving onward.}}

Long explanation:\+
\begin{DoxyItemize}
\item Dear Im\+Gui\textquotesingle{}s job is to create "{}meshes"{}, defined in a renderer-\/agnostic format made of draw commands and vertices. At the end of the frame, those meshes (\doxylink{struct_im_draw_list}{Im\+Draw\+List}) will be displayed by your rendering function. They are made up of textured polygons and the code to render them is generally fairly short (a few dozen lines). In the examples/\+ folder, we provide functions for popular graphics APIs (Open\+GL, DirectX, etc.).
\item Each rendering function decides on a data type to represent "{}textures"{}. The concept of what is a "{}texture"{} is entirely tied to your underlying engine/\+graphics API. We carry the information to identify a "{}texture"{} in the Im\+Texture\+ID type, which itself tends to be stored inside a \doxylink{struct_im_texture_ref}{Im\+Texture\+Ref}. Im\+Texture\+ID default to Im\+U64 aka 8 bytes worth of data:\+ just enough to store one pointer or integer of your choice. Dear Im\+Gui doesn\textquotesingle{}t know or understand what you are storing in Im\+Texture\+ID, it merely passes values until they reach your rendering function.
\item In the \href{https://github.com/ocornut/imgui/tree/master/examples}{\texttt{examples/\+}} backends, for each graphics API we decided on a type that is likely to be a good representation for specifying an image from the end-\/user perspective. Please bear in mind that using {\ttfamily std::\+string} on applications with a large amount of UI may incur unsatisfactory performances. Modern implementations of {\ttfamily std::\+string} often include small-\/string optimization (which is often a local buffer) but those are not configurable and not the same across implementations.

Since 1.\+92 (June 2025) fonts may be dynamically used at any size.

{\bfseries{Scaling fonts}}

Select default size:\+ 
\begin{DoxyCode}{0}
\DoxyCodeLine{style.FontSizeBase\ =\ 20.0f;}

\end{DoxyCode}
 Scale all fonts:\+ 
\begin{DoxyCode}{0}
\DoxyCodeLine{style.FontScaleDpi\ =\ 2.0f;}

\end{DoxyCode}


To change font size:\+ 
\begin{DoxyCode}{0}
\DoxyCodeLine{ImGui::PushFont(NULL,\ 42.0f);\ \textcolor{comment}{//\ This\ will\ be\ multiplied\ by\ style.FontScaleDpi}}

\end{DoxyCode}
 To change font and font size:\+ 
\begin{DoxyCode}{0}
\DoxyCodeLine{ImGui::PushFont(new\_font,\ 42.0f);}

\end{DoxyCode}


In {\ttfamily docking} branch or with multi-\/viewports:\+ 
\begin{DoxyCode}{0}
\DoxyCodeLine{io.ConfigDpiScaleFonts\ =\ \textcolor{keyword}{true};\ \ \ \ \ \ \ \ \ \ \textcolor{comment}{//\ (Docking\ branch\ only)\ Automatically\ overwrite\ style.FontScaleDpi\ in\ Begin()\ when\ Monitor\ DPI\ changes.\ This\ will\ scale\ fonts\ but\ \_NOT\_\ scale\ sizes/padding\ for\ now.}}
\DoxyCodeLine{io.ConfigDpiScaleViewports\ =\ \textcolor{keyword}{true};\ \ \ \ \ \ \textcolor{comment}{//\ (Docking\ branch\ only)\ Scale\ Dear\ ImGui\ and\ Platform\ Windows\ when\ Monitor\ DPI\ changes.}}

\end{DoxyCode}


{\bfseries{Scaling style}} (paddings, spacings, thicknesses)

This is still massively work in progress, expect turbulence. Style values are currently not easily scalable dynamically. For single viewport application you can call once:\+ 
\begin{DoxyCode}{0}
\DoxyCodeLine{style.ScaleAllSizes(factor);\ \textcolor{comment}{//\ call\ once!}}

\end{DoxyCode}
 If you need to change the scaling factor, it is currently most practical to reset the style and call this again with a new value.

Your UI code should avoid using hardcoded constants for size and positioning. Prefer to express values as multiple of reference values such as {\ttfamily Im\+Gui::\+\+Get\+Font\+Size()} or {\ttfamily Im\+Gui::\+\+Get\+Frame\+Height()}. So e.\+g. instead of seeing a hardcoded height of 500 for a given item/\+window, you may want to use {\ttfamily 30\texorpdfstring{$\ast$}{*}\+Im\+Gui::\+\+Get\+Font\+Size()} instead.

Down the line Dear Im\+Gui will provide a variety of standardized reference values to facilitate using this. This is expected to happen during subsequent 1.\+92.\+x releases.

Applications in the {\ttfamily examples/\+} folder are partly DPI aware but they are unable to load a custom font from the file-\/system, so they look ugly (may change that in the future).

The reason DPI is not auto-\/magically solved in stock examples is that we don\textquotesingle{}t yet have a satisfying solution for the "{}multi-\/dpi"{} problem (using the {\ttfamily docking} branch:\+ when multiple viewport windows are over multiple monitors using different DPI scales) specifically for the {\ttfamily \doxylink{struct_im_gui_style}{Im\+Gui\+Style}} structure. Fonts are however now perfectly scalable.

{\bfseries{On Windows, you need to inform Windows that your application is DPI aware!}} If this is not done, Windows will scale the application window and the UI text will be blurry. Potential solutions to indicate DPI awareness on Windows are:\+
\begin{DoxyItemize}
\item For SDL2:\+ the flag {\ttfamily SDL\+\_\+\+WINDOW\+\_\+\+ALLOW\+\_\+\+HIGHDPI} needs to be passed to {\ttfamily SDL\+\_\+\+Create\+Window()} + call {\ttfamily ::\+\+Set\+Process\+DPIAware()}.
\item For SDL3:\+ the flag {\ttfamily SDL\+\_\+\+WINDOW\+\_\+\+HIGH\+\_\+\+PIXEL\+\_\+\+DENSITY} needs to be passed to {\ttfamily SDL\+\_\+\+Create\+Window()}.
\item For GLFW:\+ this is done automatically.
\item For other Windows projects with other backends, or wrapper projects:\+
\begin{DoxyItemize}
\item We provide a {\ttfamily Im\+Gui\+\_\+\+Impl\+Win32\+\_\+\+Enable\+Dpi\+Awareness()} helper method in the Win32 backend.
\item Use an \href{https://learn.microsoft.com/en-us/windows/win32/hidpi/setting-the-default-dpi-awareness-for-a-process}{\texttt{application manifest file}} to set the {\ttfamily \texorpdfstring{$<$}{<}dpi\+Aware\texorpdfstring{$>$}{>}} property.
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md__mono_engine_2thirdparty_2imgui_2docs_2_f_a_q_autotoc_md603}{}\doxysubsubsection{\texorpdfstring{Q:\+ How can I load a different font than the default?}{Q:\+ How can I load a different font than the default?}}\label{md__mono_engine_2thirdparty_2imgui_2docs_2_f_a_q_autotoc_md603}
